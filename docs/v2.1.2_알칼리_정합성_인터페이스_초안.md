# v2.1.2 알칼리 적용 정합성 인터페이스 초안

이 문서는 **구현 코드가 아니라 설계 초안**이다.

목적은 다음 두 가지다.

- 알칼리 적용 방식(`aqueous direct` vs `fresh organic saponification`)의 물리적 의미를 인터페이스 수준에서 분리
- `target pH`, `fixed NaOH`, `saponification` 경로를 하나의 일관된 데이터 모델로 재구성

---

## 1. 배경

현재 엔진은 아래 문제가 있다.

1. `target pH + saponification` 다단 경로에서 사용자가 입력한 `Q_NaOH`가 실제 다단 계산에 반영되지 않는다.
2. `raw_feed_fixed_saponification`은 실제 sap inventory 전달이 아니라 `equivalent target pH` 근사 경로에 크게 의존한다.
3. 공식 회귀 통과 기준이 `legacy_premixed_target_pH`에 치우쳐 있어, 실제 raw-feed 운전 조건과의 물리적 정합성을 충분히 보장하지 못한다.

이 문제는 속도보다 먼저, **알칼리의 물리적 의미와 검증 기준**을 바로잡아야 해결된다.

---

## 2. 설계 목표

### 2.1 필수 목표

- `aqueous_direct`와 `fresh_organic_saponification`을 서로 다른 경계조건으로 명시
- `fixed_input`과 `solve_to_target`을 서로 다른 제어 방식으로 명시
- stage별 pH 제어는 `aqueous_direct`에서만 허용
- fresh organic saponification은 stage별 독립 pH 제어 수단으로 취급하지 않음
- sap capacity를 다단 유기상과 함께 역류시키는 구조를 인터페이스에 반영
- 결과 객체에 금속 수지뿐 아니라 알칼리/sap 수지 진단값 포함

### 2.2 비목표

- 이 문서는 수치 최적화나 계산 속도 개선 스펙이 아니다.
- 활동도 모델 확대나 새로운 열역학 식 도입은 이 문서 범위 밖이다.

---

## 3. 용어 정의

### 3.1 알칼리 적용 위치

- `aqueous_direct`
  - NaOH 용액이 수계에 직접 주입된다.
  - 수계 유량 증가와 희석이 발생한다.
- `fresh_organic_saponification`
  - fresh organic이 Na-form extractant inventory를 가지고 유입된다.
  - 수계 유량은 직접 증가하지 않는다.
  - 알칼리 효과는 sap capacity 전달과 proton neutralization으로 나타난다.

### 3.2 제어 방식

- `fixed_input`
  - 사용자가 알칼리 투입량을 지정한다.
- `solve_to_target`
  - solver가 목표 변수에 맞는 알칼리 투입량을 역산한다.

### 3.3 목표 종류

- `uniform_stage_pH`
  - 모든 stage 동일 pH 목표
- `stage_pH_profile`
  - stage별 개별 pH 목표
- `raffinate_pH`
  - 최종 raffinate pH 목표
- `none`
  - 목표 없음

---

## 4. 허용 조합

| application | control | target | 허용 여부 | 이유 |
|---|---|---|---|---|
| `aqueous_direct` | `fixed_input` | `none` | 허용 | 기존 고정 NaOH 의미와 일치 |
| `aqueous_direct` | `solve_to_target` | `uniform_stage_pH` | 허용 | 각 stage에서 직접 중화 가능 |
| `aqueous_direct` | `solve_to_target` | `stage_pH_profile` | 허용 | stage별 NaOH 분배를 물리적으로 해석 가능 |
| `fresh_organic_saponification` | `fixed_input` | `none` | 허용 | 실제 pre-saponified organic feed 해석 |
| `fresh_organic_saponification` | `solve_to_target` | `raffinate_pH` | 허용 | fresh sap inventory를 역산하는 전역 제어 |
| `fresh_organic_saponification` | `solve_to_target` | `uniform_stage_pH` | 금지 | fresh organic 하나로 각 stage pH를 독립 제어할 수 없음 |
| `fresh_organic_saponification` | `solve_to_target` | `stage_pH_profile` | 금지 | 물리적으로 과도한 제어 가정 |

금지 조합은 인터페이스 계층에서 즉시 에러를 내야 한다.

```python
AlkaliApplicationMode = Literal[

AlkaliControlMode = Literal[
    "fixed_input",
    "solve_to_target",

    "stage_pH_profile",
    "raffinate_pH",

    "naoh_solution_flow",
    "fresh_sap_mol_flow",
    "fresh_saponification_fraction",
    "wtpct_and_flow",
]
```

### 5.2 stage 분배 스펙

```python
@dataclass
class StageDistributionSpec:
    mode: Literal["uniform", "front_loaded", "custom"] = "uniform"
    weights: list[float] | None = None
```

의미:

- `aqueous_direct`에서만 사용
- `fresh_organic_saponification`에는 사용하지 않음

### 5.3 aqueous direct 스펙

```python
@dataclass
class AqueousDirectAlkaliSpec:
    application: Literal["aqueous_direct"] = "aqueous_direct"
    control: AlkaliControlMode = "fixed_input"
    basis: Literal["naoh_solution_flow", "wtpct_and_flow"] = "naoh_solution_flow"
    concentration_m: float = 0.0
    solution_flow_l_hr: float = 0.0
    distribution: StageDistributionSpec | None = None
    target_kind: AlkaliTargetKind = "none"
    uniform_stage_pH: float | None = None
    stage_pH_profile: list[float] | None = None
```

규칙:

- `fixed_input`이면 `target_kind == "none"`
- `solve_to_target`이면 `target_kind`는 `uniform_stage_pH` 또는 `stage_pH_profile`
- `solve_to_target`일 때 `concentration_m`은 희석 계산용으로 필수

### 5.4 fresh organic saponification 스펙

```python
@dataclass
class FreshOrganicSaponificationSpec:
    application: Literal["fresh_organic_saponification"] = "fresh_organic_saponification"
    control: AlkaliControlMode = "fixed_input"
    basis: Literal["fresh_sap_mol_flow", "fresh_saponification_fraction", "wtpct_and_flow"] = "fresh_sap_mol_flow"
    fresh_sap_mol_flow: float | None = None
    fresh_saponification_fraction: float | None = None
    naoh_concentration_m: float | None = None
    naoh_flow_l_hr: float | None = None
    target_kind: AlkaliTargetKind = "none"
    raffinate_pH_target: float | None = None
```

규칙:

- `fixed_input`이면 `target_kind == "none"`
- `solve_to_target`이면 `target_kind == "raffinate_pH"`만 허용
- `basis == "wtpct_and_flow"`는 fresh organic sap capacity 환산용 입력일 뿐, 수계 직접 주입 의미가 아님

### 5.5 통합 입력 스펙

## 6. solver 인터페이스 초안

### 6.1 단일단 solver

```python
@dataclass
class StageState:
    C_aq_in: dict[str, float]
    C_org_in: dict[str, float]
    pH_in: float
    Q_aq_in_L_hr: float
    Q_org_in_L_hr: float
    C_sulfate_in_M: float


@dataclass
class StageAlkaliState:
    application: AlkaliApplicationMode
    aqueous_naoh_flow_l_hr: float = 0.0
    aqueous_naoh_molarity_M: float = 0.0
    saponified_extractant_in_mol_hr: float = 0.0


@dataclass
class StageObjective:
    target_kind: Literal["none", "stage_pH"] = "none"
    target_pH: float | None = None


def solve_single_stage_v2(
    *,
    stage: StageState,
    alkali_state: StageAlkaliState,
    objective: StageObjective,
    extractant: str,
    C_ext: float,
    metals: list[str],
    temperature: float | None = None,
    use_competition: bool = False,
    use_speciation: bool = False,
    extractant_params: dict | None = None,
) -> "StageResult":
    ...
```

단일단 해석 규칙:

- `aqueous_direct`
  - `aqueous_naoh_flow_l_hr`와 `aqueous_naoh_molarity_M`를 사용
  - `Q_aq_out = Q_aq_in + aqueous_naoh_flow_l_hr`
- `fresh_organic_saponification`
  - `saponified_extractant_in_mol_hr`를 사용
  - `Q_aq_out = Q_aq_in`
- `objective.target_kind == "stage_pH"`는 `aqueous_direct`에서만 허용

### 6.2 다단 solver

```python
@dataclass
class CascadeObjective:
    target_kind: AlkaliTargetKind = "none"
    uniform_stage_pH: float | None = None
    stage_pH_profile: list[float] | None = None
    raffinate_pH_target: float | None = None


def solve_multistage_countercurrent_v2(
    *,
    spec: SimulationSpec,
    objective: CascadeObjective | None = None,
) -> "CascadeResult":
    ...
```

다단 해석 규칙:

- `aqueous_direct + fixed_input`
  - 총 NaOH 유량을 stage distribution에 따라 분배
- `aqueous_direct + solve_to_target`
  - stage별 direct NaOH를 역산해 목표 pH 달성
- `fresh_organic_saponification + fixed_input`
  - fresh organic inlet에 sap inventory를 부여
  - stage `N`의 sap out이 stage `N-1`의 sap in이 되도록 역류 전달
- `fresh_organic_saponification + solve_to_target`
  - root variable은 `fresh_sap_mol_flow`
  - objective는 `raffinate_pH_target`
  - stage별 pH target을 직접 두지 않음

## 8. 내부 계산 계약

### 8.1 aqueous direct 계약

1. stage 입력:
   - `Q_aq_in`
   - `C_aq_in`
   - `aqueous_naoh_flow_l_hr`
   - `aqueous_naoh_molarity_M`
2. stage 출력:
   - `Q_aq_out = Q_aq_in + aqueous_naoh_flow_l_hr`
   - 금속 농도는 증가한 유량 기준으로 환산
   - proton balance는 실제 투입 OH 기준으로 계산

### 8.2 fresh organic saponification 계약

1. cascade fresh organic inlet에서만 sap inventory가 새로 들어온다.
2. 각 stage는
   - `saponified_extractant_in_mol_hr`
   - `C_org_in`
   를 입력으로 받는다.
3. 각 stage는 금속 교환과 직접 중화에 사용된 sap를 차감해
   - `saponified_extractant_out_mol_hr`
   를 반환한다.
4. 이 값은 한 단계 앞 stage의 organic-side alkali input이 된다.
5. `Q_aq`는 직접 증가하지 않는다.

### 8.3 target solving 계약

- `aqueous_direct + stage_pH target`
  - stage별 필요한 `aqueous_naoh_flow_l_hr`를 푼다.
- `fresh_organic_saponification + raffinate_pH target`
  - cascade 전체를 한 번 계산한 뒤 최종 raffinate pH residual을 본다.
  - root solve variable은 `fresh_sap_mol_flow`

## 9. 하위호환 전략

기존 공개 함수는 당장 제거하지 않는다.

### 9.1 유지할 함수

- `solve_single_stage(...)`
- `solve_multistage_countercurrent(...)`

### 9.2 어댑터 계층

기존 kwargs를 새 스펙으로 변환하는 adapter를 둔다.

```python
def build_simulation_spec_from_legacy_kwargs(**kwargs) -> tuple[SimulationSpec, CascadeObjective | None]:
    ...
```

변환 규칙:

- `target_pH` + `naoh_mode="aqueous_direct"`:
  - `AqueousDirectAlkaliSpec(control="solve_to_target", target_kind="uniform_stage_pH")`
- `target_pH_per_stage` + `naoh_mode="aqueous_direct"`:
  - `AqueousDirectAlkaliSpec(control="solve_to_target", target_kind="stage_pH_profile")`
- `Q_NaOH` + `naoh_mode="saponification"` + `target_pH is None`:
  - `FreshOrganicSaponificationSpec(control="fixed_input")`
- `target_pH` + `naoh_mode="saponification"`:
  - 기본은 `ValueError`
  - 단, 명시적 `legacy_allow_equivalent_target_pH=True`일 때만 surrogate fallback 허용

## 10. UI 인터페이스 초안

### 10.1 상위 진입은 과업 중심으로 둔다

연구원은 내부 enum보다 **무슨 작업을 하려는지**로 접근하는 편이 자연스럽다.

권장 상위 진입:

1. `현장 replay`
   - 현재 프리셋/실측 운전 재현
   - 필요 시 legacy fallback 경로 허용
2. `수계 direct pH 설계`
   - 목표 pH 또는 stage별 pH profile 설계
   - `aqueous_direct`로 고정
3. `fresh organic 사포니피케이션 설계`
   - fresh sap inventory 지정 또는 최종 raffinate pH 역산
   - `fresh_organic_saponification`로 고정

### 10.2 고급 설정에서만 내부 모델 축을 노출한다

고급 설정 펼침 영역에서 아래를 보여준다.

1. `알칼리 적용 위치`
   - 수계 직접 주입
   - fresh organic 사포니피케이션
2. `제어 방식`
   - 고정 입력
   - 목표값 역산
3. `목표 종류`
   - 없음
   - 모든 stage 동일 pH
   - stage별 pH profile
   - 최종 raffinate pH

즉 기본 사용자는 과업 중심으로 시작하고, 고급 사용자가 필요할 때만 내부 모델 축을 직접 만진다.

### 10.3 UI 제약

- `fresh organic 사포니피케이션` 선택 시:
  - `Stage별 차등 pH` 옵션을 숨긴다.
  - 목표 모드는 `최종 raffinate pH 목표`만 표시한다.
  - 추출제 비교 탭도 `비교 목표 pH` 대신 `비교 fresh sap` 또는 `비교 raffinate pH` 문맥으로 바꾼다.
- `수계 직접 주입` 선택 시:
  - `uniform stage pH` 또는 `stage별 pH profile`을 허용한다.
  - NaOH 희석 계산 옵션을 노출한다.
- 금지 조합은 단순 에러보다 먼저 UI에서 **선택 불가/숨김**으로 처리한다.
- 현장 프리셋은 기존과 동일하게 one-click replay가 되도록 하고, 고급 설정은 접힌 상태를 기본으로 둔다.

### 10.4 표시해야 할 결과

- fresh sap input
- stage별 sap out
- stage별 metal exchange consumption
- stage별 direct neutralization
- total effective alkalinity
- 현재 결과가 `legacy fallback`인지, `physical sap cascade`인지 명시
- 현재 검증 기준이 `release gate`인지, `shadow diagnostic`인지 명시

### 10.5 사용성 수칙

- 현재 기본 사용자 흐름보다 입력 단계가 늘어나지 않도록 한다.
- 현장 replay는 프리셋 선택 후 추가 1~2개 입력 안에서 끝나야 한다.
- 초심자가 `application/control/target` 조합표를 직접 이해하지 않아도 시뮬레이션을 돌릴 수 있어야 한다.

## 11. 검증 데이터 메타데이터 및 승격 게이트 초안

현장 데이터셋별로 아래 메타데이터를 추가한다.

```python
@dataclass
class ValidationCaseMeta:
    input_basis: Literal["raw_feed", "premixed_feed"]
    alkali_application: Literal["aqueous_direct", "fresh_organic_saponification", "unknown"]
    alkali_control_mode: Literal["fixed_input", "solve_to_target", "unknown"]
    pH_measurement_location: Literal["feed", "stage_outlet", "raffinate", "unknown"]
    measured_raffinate_pH: float | None

    naoh_concentration_m: float | None
    naoh_concentration_source: Literal["measured", "derived_from_wtpct", "assumed", "unknown"]
    naoh_flow_source: Literal["measured", "assumed", "unknown"]

    stagewise_distribution_known: bool
    injection_stage_indices: list[int] | None
    distribution_weights: list[float] | None

    fresh_organic_sap_measured: bool
    can_be_primary_pass_fail: bool
```

승격 기준:

- 아래 조건을 동시에 만족해야 `raw_feed`를 1차 pass/fail 기준으로 승격한다.
  - `alkali_application`이 명확하다.
  - `naoh_concentration_m`가 실측 또는 신뢰 가능한 파생값이다.
  - 주입 위치/분배 정보가 알려져 있다.
  - pH 측정 위치가 해석 대상과 일치한다.
- 위 조건이 하나라도 빠지면:
  - `raw_feed`는 shadow diagnostic으로만 사용
  - `premixed_feed + target_pH replay`는 release gate로 유지

### 11.1 현재 단계의 판단

지금 당장은 `raw_feed + measured alkali mode` 승격을 선언하는 것이 아니라, **승격을 가능하게 하는 메타데이터 조건을 먼저 문서화**하는 단계로 본다.

## 12. 병행 회귀 및 단계적 롤아웃 초안

새 물리 경로를 넣더라도 기존 경로를 즉시 제거하지 않는다.

### 12.1 동시에 유지할 세 가지 경로

1. `legacy_premixed_target_pH`
   - 현재 release gate
2. `raw_feed_fixed_saponification_legacy`
   - 현재 raw-feed replay 기준선
   - field-calibrated fallback shadow reference
3. `raw_feed_physical_saponification_v2`
   - 새 물리 경로
   - 초기에 experimental / shadow mode

### 12.2 승격 규칙

- `raw_feed_physical_saponification_v2`가 아래를 만족하기 전까지는 release gate로 승격하지 않는다.
  - Cyanex 272 field set에서 `raw_feed_fixed_saponification_legacy`보다 aggregate MAE가 나쁘지 않다.
  - D2EHPA field set에서도 동일 조건을 만족한다.
  - dataset별 worst error가 특정 금속에서 체계적으로 악화되지 않는다.
  - random multistage convergence와 runtime이 허용 범위 안이다.
- `equivalent target pH` fallback은 위 조건 충족 전까지는 제거하지 않고, 명시적 legacy path로 유지한다.

### 12.3 보고 방식

- release gate 결과와 shadow diagnostic 결과를 나란히 기록한다.
- 문서/대시보드에서 `legacy fallback`과 `physical v2`를 혼동하지 않게 이름을 분리한다.

## 13. 수용 기준 초안

### 13.1 인터페이스 수용 기준

- 금지 조합은 생성 시점에 명시적으로 실패하거나 UI에서 선택 불가 처리
- `target pH + saponification` 다단에서 입력 `Q_NaOH`가 결과에 무영향인 legacy 조합은 기본 UI에서 노출하지 않음
- fresh sap inventory는 다단에서 stage 간 counter-current로 전달돼야 함
- 결과 객체에 metal / proton / sap inventory diagnostics가 모두 포함돼야 함

### 13.2 정확도 수용 기준

- 정확도 평가는 기본적으로 **실험값 대비 상대오차 `%`**를 1차 지표로 사용한다.
- 단, 실험 후액값이 아주 작은 trace 영역(`예: < 0.1 g/L`)은 상대오차가 불안정하므로 절대오차 `g/L`를 보조 지표로 사용한다.
- single-stage metal balance error: 각 금속 `<= 1e-6 ~ 1e-4 g` 수준
- cascade metal balance error: 각 금속 `<= 1e-4 g`
- sap inventory balance error threshold 별도 정의
- `raw_feed_physical_saponification_v2`는 용매별로 아래 조건을 모두 만족해야 함
  - Cyanex 272 field aggregate relative error `%`가 현재 `raw_feed_fixed_saponification_legacy`보다 나쁘지 않음
  - D2EHPA field aggregate relative error `%`가 현재 `raw_feed_fixed_saponification_legacy`보다 나쁘지 않음
  - D2EHPA에서 핵심 KPI인 `Ni raffinate` relative error `%`가 체계적으로 악화되지 않음
  - Cyanex 272에서 `Li raffinate` relative error `%`가 체계적으로 악화되지 않음
  - trace 영역 금속은 절대오차 `g/L`도 함께 확인
  - 특정 dataset 한두 개만 좋아지고 전체 평균이 나빠지는 local fit이면 통과로 간주하지 않음

### 13.3 강건성 수용 기준

- 동일 random seed 세트에서 용매별 결과를 분리 집계한다.
- 최소 집계 축:
  - Cyanex 272 / D2EHPA
  - `aqueous_direct fixed_input`
  - `aqueous_direct solve_to_target`
  - `fresh_organic_saponification fixed_input`
  - `fresh_organic_saponification solve_to_target`
- 각 축에서 아래를 기록한다.
  - convergence rate
  - mass balance fail count
  - non-negative 위반 count
  - p95 iteration
  - p95 runtime
- 새 경로는 대응되는 legacy 경로보다 failure count를 늘리면 release gate 통과로 보지 않는다.

### 13.4 사용성 수용 기준

- 현장 replay 기본 흐름의 입력 개수는 현재 대비 증가하지 않는다.
- 연구원이 가장 자주 쓰는 목적:
  - 현장 replay
  - 조건 설계
  - 추출제 비교
  에서 첫 실행까지의 클릭 수가 크게 늘지 않아야 한다.
- 추출제 비교 탭은 선택된 알칼리 문맥을 따라야 한다.
  - `aqueous_direct`면 pH 비교
  - `fresh_organic_saponification`이면 sap 또는 raffinate pH 비교
- 결과 화면에 현재 계산이 `실측 기반`, `가정 기반`, `legacy fallback` 중 무엇인지 항상 보여줘야 한다.

### 13.5 회귀 테스트 초안

1. `aqueous_direct fixed_input` replay
2. `aqueous_direct solve_to_target uniform_stage_pH`
3. `aqueous_direct solve_to_target stage_pH_profile`
4. `fresh_organic_saponification fixed_input`
5. `fresh_organic_saponification solve_to_target raffinate_pH`
6. 금지 조합 validation
7. stage/cascade metal balance
8. proton/sap inventory balance
9. 용매별 field MAE split
10. 용매별 random multistage stress
11. `target pH + saponification` legacy 무감도 regression

## 14. 정확도 향상 관점에서 이 초안의 역할

이 초안은 **정확도 향상에 필요한 선행 조건**을 정리한 것이다.

### 14.1 이 초안으로 직접 개선되는 것

- 잘못된 제어 조합 때문에 생기던 해석 오류 제거
- `target pH + saponification`에서 입력 `Q_NaOH`가 무의미해지는 문제 제거
- `fresh sap inventory`를 실제 counter-current state로 다룰 수 있게 됨
- raw-feed 검증과 legacy replay 검증을 섞어 보던 문제 완화

즉 현재 정확도 저하 중 **알칼리 해석 오류에서 오는 부분**은 이 초안으로 직접 개선 대상이 된다.

### 14.2 이 초안만으로는 해결되지 않는 것

- 실측 NaOH 농도/주입 위치 메타데이터 부족
- D2EHPA 고황산염 구간의 잔여 오차
- Cyanex 272 / D2EHPA에서 남아 있는 Li, Co 편차
- 추출제 파라미터와 종분화/경쟁추출 모델 자체의 잔여 오차

즉 사용자가 말한 의미의 **현장 실험 대비 정확도 향상**을 위해서는,
이 초안 구현만으로 끝나지 않고 아래 후속 작업이 필요하다.

1. dataset 메타데이터 보강
2. 새 물리 경로 구현
3. 상대오차 `%` 기준 field 회귀 재평가
4. 남는 편차에 대한 화학 모델/파라미터 보정

결론적으로 이 초안은 **필요하지만 충분조건은 아니다**.

### 15.1 field 검증

### 15.3 방향성 판단 기준

- field에서 `raw_feed_target_pH`보다 `raw_feed_fixed_saponification`이 일관되게 낫다면,
  - fresh organic saponification을 `stage별 pH 제어`가 아니라 `sap inventory 문제`로 다루는 방향이 맞다.
- random에서 `target pH + saponification`이 `Q_NaOH` 변화에 둔감하거나 무감하면,
  - 해당 조합을 기본 UI에서 제거하는 방향이 맞다.
- D2EHPA에서만 취약성이 크면,
  - 용매별 승격 기준을 분리해 관리하는 편이 맞다.

## 17. 결정 사항 요약

- `fresh organic saponification`은 `stage별 pH 제어 수단`이 아니다.
- `target pH + saponification`은 기본적으로 `raffinate pH target`으로만 해석한다.
- fixed saponification은 fresh sap inventory를 실제로 역류 전달하는 구조로 바꾼다.
- `equivalent target pH` 회귀식은 즉시 제거하지 않고, 검증 완료 전까지 legacy fallback으로 유지한다.
- 공식 정확도 평가는 단일 기준이 아니라 `legacy release gate + raw-feed shadow + v2 experimental` 병행 체계로 운영한다.
- 용매별로 `Cyanex 272`와 `D2EHPA`를 분리해 승격 여부를 판단한다.
