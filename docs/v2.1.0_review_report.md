# v2.1.0 Review Report

- Review date: 2026-03-07
- Reviewed commit: `327aaa6` (`v2.1.0: 엔진 정밀화, 고정 NaOH 수렴 개선, 코드 품질 및 UI 개선`)
- Scope: simulation engine, built-in validation scripts, `docs/` claims, Streamlit dashboard behavior

## Verdict

The app is runnable and the core solver remains structurally sound. Mass balance and multistage convergence checks pass, and the Streamlit dashboard starts successfully.

The main remaining risk is not "the app is broken", but "the current documentation and validation claims overstate how well the updated model is verified". In particular, convergence and mass balance are strong, but field-fit accuracy is still uneven for some datasets, and several `v2.1.0` claims are not backed by committed, reproducible regression code.

## Checks Performed

1. Ran `validation_test.py` against current HEAD.
2. Ran `test_verification.py` against current HEAD.
3. Re-ran the built-in field datasets with the same sulfate calculation used by the dashboard instead of the hardcoded approximation in `test_verification.py`.
4. Verified that `sx_dashboard.py` launches under Streamlit without runtime exceptions.
5. Compared `v2.1.0` docs claims against the actual code and the committed test coverage.

## Findings

### 1. High: `ALL PASS` in docs currently means "solver converged and conserved mass", not "field accuracy is good enough"

Evidence:
- [`docs/v2.1.0_improvement_plan.md:4`](./v2.1.0_improvement_plan.md) says `전체 적용 완료 + 회귀 테스트 ALL PASS`.
- [`docs/v2.1.0_improvement_plan.md:546`](./v2.1.0_improvement_plan.md) to [`docs/v2.1.0_improvement_plan.md:564`](./v2.1.0_improvement_plan.md) reports final regression success.
- Current dashboard-equivalent rerun still shows large field residuals for some datasets:
  - Data1 Ni: `0.769` vs experiment `7.166` (`-6.397 g/L`)
  - Data2 Ni: `5.955` vs `8.951` (`-2.996 g/L`)
  - Data4 Li: `8.495` vs `6.335` (`+2.160 g/L`)
  - Data6 Co: `0.003` vs `0.848` (`-0.845 g/L`)

Why this matters:
- A reader can reasonably interpret `ALL PASS` as "the updated model is validated well enough against field data".
- The committed evidence does support solver stability and mass balance, but it does not support a blanket field-accuracy pass.

Recommendation:
- Split reporting into two lines:
  - solver integrity: PASS
  - field-fit accuracy: mixed / condition-dependent

### 2. High: the committed field-validation script does not reproduce actual dashboard conditions

Evidence:
- [`test_verification.py:77`](../test_verification.py#L77) hardcodes `C_sulfate=0.3`.
- The actual dashboard computes sulfate from feed composition at [`sx_dashboard.py:130`](../sx_dashboard.py#L130) to [`sx_dashboard.py:139`](../sx_dashboard.py#L139).

Why this matters:
- `test_verification.py` is currently the closest thing to a field-regression script, but it is not running the same chemistry inputs the dashboard uses.
- That makes the doc claims harder to trust, even when the qualitative result trend remains similar.

Recommendation:
- Replace the hardcoded sulfate in `test_verification.py` with the same formula used in the dashboard.
- Keep one validation path only, so docs, scripts, and UI all reference the same calculation.

### 3. High: the new `v2.1.0` claims are not backed by committed reproducible regression tests for the changed branches

Evidence:
- The `v2.1.0` commit changed:
  - fixed-NaOH internals in [`sx_simulator/single_stage.py:238`](../sx_simulator/single_stage.py#L238)
  - multistage relaxation in [`sx_simulator/multistage_sx.py:124`](../sx_simulator/multistage_sx.py#L124)
  - weighted `n_ext` logic in [`sx_simulator/extraction_isotherm.py:351`](../sx_simulator/extraction_isotherm.py#L351)
  - compare-tab simulation inputs in [`sx_dashboard.py:594`](../sx_dashboard.py#L594)
- But the committed tests are still:
  - [`validation_test.py`](../validation_test.py), which only checks target-pH mode, a Co isotherm sanity check, and one cascade mass-balance case
  - [`test_verification.py`](../test_verification.py), which prints differences but has no thresholds and no failure conditions
- The fixed-NaOH regression claim in [`docs/v2.1.0_improvement_plan.md:559`](./v2.1.0_improvement_plan.md#L559) to [`docs/v2.1.0_improvement_plan.md:560`](./v2.1.0_improvement_plan.md#L560) is not traceable to a committed script.

Why this matters:
- The modified code paths can regress silently.
- The docs are currently stronger than the automated evidence in the repo.

Recommendation:
- Add a committed regression script or test module for:
  - fixed NaOH mode convergence
  - weighted `n_ext` behavior
  - compare-tab parity with the main simulation inputs
- Make the script exit non-zero when thresholds are violated.

### 4. Medium: the "generalization" report overstates how little tuning the model contains

Evidence:
- [`docs/v2.1.0_모델_일반화_검토보고서.md:21`](./v2.1.0_모델_일반화_검토보고서.md#L21) says overfit risk is low because only `2` of `49` parameters were adjusted.
- The same document later uses `126` total parameters at [`docs/v2.1.0_모델_일반화_검토보고서.md:57`](./v2.1.0_모델_일반화_검토보고서.md#L57) to [`docs/v2.1.0_모델_일반화_검토보고서.md:63`](./v2.1.0_모델_일반화_검토보고서.md#L63), so the report is internally inconsistent.
- More importantly, the solver also contains phenomenological competition heuristics that materially affect behavior, especially in high-loading cases:
  - `n_eff_multiplier = max(0.35, 1.0 - (loading_ratio_clamped ** 2))` at [`sx_simulator/extraction_isotherm.py:373`](../sx_simulator/extraction_isotherm.py#L373)
  - weighted `n_ext` logic at [`sx_simulator/extraction_isotherm.py:354`](../sx_simulator/extraction_isotherm.py#L354) to [`sx_simulator/extraction_isotherm.py:365`](../sx_simulator/extraction_isotherm.py#L365)

Why this matters:
- The report frames the model as if only two scalar site adjustments exist.
- In practice, there is also embedded heuristic structure in the competition model, and that affects transferability and generalization claims.

Recommendation:
- Rephrase the document to say:
  - only two explicit `pH50` values were site-calibrated
  - but the engine also relies on heuristic competition/loading functions that were chosen for empirical performance
- Remove the contradictory parameter counts.

### 5. Medium: one field-validation dataset is internally inconsistent and is silently skipped

Evidence:
- [`test_verification.py:40`](../test_verification.py#L40) to [`test_verification.py:44`](../test_verification.py#L44) defines Data5 with Li input `0.013 g/L` and expected Li raffinate `5.400 g/L`.
- [`test_verification.py:97`](../test_verification.py#L97) to [`test_verification.py:98`](../test_verification.py#L98) skips that Li row entirely.

Why this matters:
- A validation dataset where the expected outlet exceeds the inlet by orders of magnitude is not physically self-consistent.
- Skipping it in code is pragmatic, but it weakens the credibility of the dataset unless the reason is documented explicitly.

Recommendation:
- Correct the source data or mark the row as invalid in the dataset definition itself.
- Document why Data5 Li is excluded from validation summaries.

### 6. Low: the dashboard still relies on mutable module-level parameters for interactive edits

Evidence:
- [`sx_dashboard.py:233`](../sx_dashboard.py#L233) to [`sx_dashboard.py:237`](../sx_dashboard.py#L237) still writes into `EXTRACTANT_PARAMS`.
- `v2.1.0` only adds commentary around the behavior; it does not isolate simulation calls from the mutated module state.

Why this matters:
- This is probably acceptable in the current single-user Streamlit session model.
- It is still a maintenance risk because later code may assume `EXTRACTANT_PARAMS` is pristine.

Recommendation:
- Not a release blocker for the current desktop/internal use case.
- Worth isolating later if multi-session deployment or more complex state flows are planned.

## Recommendation For Execution Decision

### Safe to proceed if:

- the goal is internal engineering use
- the goal is trend analysis, sensitivity analysis, or comparing operating conditions
- the team understands that field-fit is still mixed across datasets

### Do not present as fully validated if:

- the output will be used as a generalized predictor across other plants or solvent systems
- the output will be used to claim field-grade accuracy from the current regression evidence alone
- the `v2.1.0` docs will be treated as auditable proof of complete validation

## Suggested Next Actions Before Broader Rollout

1. Make `test_verification.py` reproduce dashboard inputs exactly.
2. Add executable regression coverage for the actual `v2.1.0` changes.
3. Downgrade `ALL PASS` wording in docs to something narrower unless field-threshold assertions are added.
4. Clean or explicitly exclude invalid validation rows such as Data5 Li.

## Bottom Line

I would approve this version for internal exploratory use.

I would not approve the current docs wording as evidence that the updated model is fully validated or broadly generalized. The code is in a better state than the documentation implies, but the documentation is currently more confident than the committed validation evidence.
