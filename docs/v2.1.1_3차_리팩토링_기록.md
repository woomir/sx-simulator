# v2.1.1 3차 리팩토링 기록

작성일: 2026-03-08

## 목적

3차 리팩토링의 목적은 `sx_dashboard.py`에 섞여 있던 입력 조립, 시뮬레이션 실행, 검증 커버리지 평가 로직을 별도 서비스 계층으로 분리하는 것이다.

이번 단계는 다음 두 문제를 줄이는 데 초점을 맞췄다.

- 대시보드 스크립트가 UI와 비즈니스 로직을 동시에 담당하던 구조
- 메인 계산과 추출제 비교 계산이 각각 별도 위치에서 수동으로 조립되던 중복 구조

## 변경 파일

- `sx_simulator/dashboard_service.py` 신규 추가
- `sx_dashboard.py` 수정
- `PROJECT_STRUCTURE.md` 신규 추가

## 주요 변경 내용

### 1. 대시보드 서비스 모듈 추가

`sx_simulator/dashboard_service.py`를 새로 만들고 다음 책임을 옮겼다.

- `SimulationInputs`
  - 현재 대시보드 입력을 한 객체로 묶는 자료형
- `build_simulation_kwargs()`
  - 엔진 호출용 kwargs 조립
- `run_simulation()`
  - 메인 시뮬레이션 실행
- `run_compare_simulations()`
  - 비교 탭용 Cyanex 272 / D2EHPA 공통 실행
- `compute_loading_pct()`
  - 결과에서 최대 로딩률 계산
- `build_scope_assessment()`
  - 현재 입력과 결과를 바탕으로 검증 커버리지 평가

이렇게 해서 시뮬레이션 조립 로직이 UI 코드에서 분리되었다.

### 2. 대시보드에서 서비스 계층 사용

`sx_dashboard.py`는 더 이상 메인 계산 kwargs를 직접 조립하지 않는다.

변경 후 흐름:

1. 사이드바에서 입력값 수집
2. `SimulationInputs` 생성
3. `run_simulation()` 호출
4. `build_scope_assessment()`로 해석 가이드 생성

추출제 비교 탭도 동일한 입력 객체를 기반으로 `run_compare_simulations()`를 호출하도록 통합했다.

### 3. 숨은 전역 의존성 일부 제거

대시보드 내부에서 탭 간 순서 의존으로 쓰이던 값들을 상수로 분리했다.

- `METAL_COLORS`
- `PH_PLOT_RANGE`

이 변경으로 특정 탭이 앞에서 정의된 지역 변수에 기대는 구조를 줄였다.

### 4. 프로젝트 구조 설명 파일 추가

루트에 `PROJECT_STRUCTURE.md`를 추가했다.

이 파일은 다음 내용을 한 번에 설명한다.

- 프로젝트 목적
- 실행 흐름
- 폴더/파일 역할
- 검증 진입점
- 현재 모델 한계

즉, 이후 코드 수정 전에 가장 먼저 읽어야 할 안내 문서 역할을 한다.

## 동작 보존 원칙

이번 리팩토링에서 바꾸지 않은 항목:

- 시그모이드 추출 모델
- 온도/농도 보정식
- 경쟁 추출 로직
- 종분화 로직
- 단일단/다단 수치해법
- 사용자 입력 항목과 탭 구성

이번 단계도 구조 정리만 수행했고, 모델 거동은 유지하는 것을 원칙으로 했다.

## 현재 기준에서 좋아진 점

- 대시보드가 시뮬레이션 조립 상세를 직접 알 필요가 줄었다.
- 메인 계산과 비교 계산이 같은 입력 구조를 재사용한다.
- 이후 탭 분리 리팩토링을 할 때 기반이 생겼다.
- 새 개발자가 어디서 입력이 엔진으로 들어가는지 더 빨리 파악할 수 있다.
