# 시뮬레이션 프로그램 제작 워크플로우 가이드

**버전**: 1.0
**작성일**: 2026-03-17
**성격**: 산업 표준 + 실전 경험 기반 방법론 문서
**범위**: 물리/화학/공학 시뮬레이션 소프트웨어 개발 전반

---

## 1. 서문: 왜 시뮬레이션 소프트웨어는 다른가

시뮬레이션 소프트웨어는 일반 소프트웨어와 근본적으로 다른 목표를 갖는다. 일반 소프트웨어는 기능 사양을 충족하면 "완성"이지만, 시뮬레이션 소프트웨어는 **물리적 현실을 정확하게 반영**해야 한다. 더 위험한 것은 시뮬레이션이 "그럴듯하게 틀릴 수 있다"는 점이다 — 숫자가 나오니까 맞다고 착각하지만, 실제로는 물리적으로 의미 없는 결과일 수 있다.

이 문제를 인식한 산업계는 NASA-STD-7009B(모델/시뮬레이션 신뢰성 표준), ASME V&V 10(고체역학 V&V), ASME V&V 20(유체/열전달 V&V) 같은 검증 프레임워크를 발전시켜 왔다. 이 표준들의 핵심 메시지는 하나다: **"시뮬레이션 결과의 신뢰성은 코드 품질이 아니라 검증 증거의 축적으로 결정된다."**

본 가이드는 이러한 산업 표준과 실전 경험을 결합하여 **"Agile + V&V Gates" 하이브리드** 접근법을 제안한다. 빠른 반복 개발(Agile)로 생산성을 확보하되, 각 단계마다 반드시 통과해야 할 검증 관문(Quality Gate)을 두어 "동작하지만 틀린" 코드가 다음 단계로 넘어가는 것을 방지한다.

이 가이드에서 반복적으로 참조하는 실전 사례는 황산염계 용매추출(SX) 공정 시뮬레이터 프로젝트다. 3주 동안 v1.0에서 v2.1.2까지 14개 버전, 5,600줄, 8개 모듈로 성장하면서 얻은 교훈이 이 가이드의 뼈대를 이룬다.

## 2. 전체 워크플로우 개관

### 9단계 다이어그램

```
Phase 0       Phase 1       Phase 2       Phase 3
문제 정의 ──→ 모델 설계 ──→ 프로토타입 ──→ 엔진 구현
    │              ↑              ↑             │
    │              └──── V&V 실패 시 회귀 ───────┤
    ↓                                           ↓
Phase 8       Phase 7       Phase 6       Phase 4
유지보수 ←── 견고성 ←───── 보정 ←─────── V&V
                                           │
                                    Phase 5 ↓
                                      UI/시각화
```

각 단계는 선형이 아니라 **피드백 루프**를 갖는다. V&V(Phase 4)에서 실패하면 모델 설계(Phase 1)로 회귀할 수 있고, 견고성 테스트(Phase 7)에서 새로운 물리 메커니즘이 드러나면 엔진(Phase 3)을 확장해야 한다.

**핵심 원칙: 점진적 복잡도 증가(Incremental Complexity)**. 가장 단순한 모델로 시작하여, 검증 데이터가 요구할 때만 복잡도를 추가한다. "나중에 필요할 것 같아서" 미리 추가하는 기능은 검증 부담만 늘리고 디버깅을 어렵게 만든다.

**Quality Gate 개념**: 각 Phase를 완료하기 전, 정량적이고 측정 가능한 기준을 통과해야 다음 단계로 진행한다. "대충 맞는 것 같다"는 주관적 판단이 아니라 "MAE < 0.5 g/L"과 같은 숫자로 정의한다.

## 3. Phase 0: 문제 정의와 범위 설정

### 일반론

시뮬레이션의 목적을 **한 문장으로** 정의하는 것에서 시작한다. "이 시뮬레이터는 무엇을 위해 존재하는가?" — 의사결정 지원인지, 설계 최적화인지, 교육용인지에 따라 정확도 요구사항과 모델 복잡도가 완전히 달라진다.

#### 입력 데이터 품질 정의 (NASA-STD-7009B "Input Pedigree")

NASA-STD-7009B의 신뢰성 8대 요소 중 첫 번째는 **Input Pedigree**다. 모든 입력 데이터의 출처와 품질을 사전에 정의해야 한다:

- **데이터 출처**: 직접 실험, 문헌, 상용 DB, 전문가 추정
- **불확실성 범위**: 측정 오차, 재현성, 대표성
- **적용 가능 조건**: 어떤 온도/압력/농도 범위에서 유효한 데이터인지

#### 정량적 정확도 요구사항

"정확해야 한다"는 요구사항은 의미가 없다. 다음을 숫자로 명시해야 한다:

- **절대오차 허용 범위**: 예) 후액 농도 오차 ±1.0 g/L 이내
- **상대오차 허용 범위**: 예) 주요 금속 추출률 상대오차 15% 이내
- **불확실성 범위**: 예) 95% 신뢰구간 내에서 실험값 포함

#### 검증 데이터 확보 전략

모델을 만들기 **전에** 검증에 사용할 데이터를 확보해야 한다:

- **실험 데이터**: 최소 3세트 이상, 서로 다른 조건
- **해석해(Analytical Solution)**: 단순화된 조건에서의 이론적 해
- **상용 소프트웨어 벤치마크**: ASPEN, MetSIM, OLI 등의 결과
- **문헌값**: 피어리뷰 논문의 보고된 결과

#### 운전/적용 조건 범위 (Operational Envelope)

시뮬레이션이 **유효한 범위**를 명시적으로 정의한다:

- 입력 변수별 최소/최대값
- 조합 조건의 제한 (예: 고농도 + 고온 동시 조건)
- 시뮬레이션이 하지 **않는** 것을 선언

### SX 프로젝트 사례

SX 시뮬레이터의 `VALIDATED_FIELD_WINDOW`는 프로젝트 후반에야 뒤늦게 정의되었다. pH 3.9~7.0, 추출제 농도 0.6053~0.6308 M, 25±5°C, 최대 5단이라는 검증 범위가 코드에 명시된 것은 v2.1.1 이후였다. 만약 이 범위를 Phase 0에서 정의했다면, Data5(Ni 71g/L)의 극단 조건이 초기 모델의 범위를 넘어선다는 것을 미리 인식하고, 모델 전면 개선(v2.0의 다핵착물 알고리즘)을 보다 계획적으로 진행할 수 있었을 것이다.

### Quality Gate

- [ ] 검증 데이터 ≥ 3세트 확보 완료
- [ ] 정확도 기준을 절대오차/상대오차/worst-case로 숫자화
- [ ] 운전 조건 범위(Operational Envelope) 문서화 완료
- [ ] 시뮬레이션의 한계를 "무엇을 하지 않는가"로 명시

## 4. Phase 1: 수학 모델 설계

### 일반론

#### 모델 계층 결정 기준 7가지

시뮬레이션 모델을 설계할 때, 다음 7가지 기준을 종합적으로 평가하여 적절한 모델 계층을 선택한다:

| # | 기준 | 설명 | 트레이드오프 |
|---|------|------|-------------|
| 1 | **물리적 충실도** | 실제 현상을 얼마나 정확하게 반영하는가 | 높을수록 파라미터 ↑, 계산 비용 ↑ |
| 2 | **계산 비용** | 1회 시뮬레이션에 걸리는 시간과 자원 | 실시간 UI 필요 시 0.1초 이내 요구 |
| 3 | **파라미터 가용성** | 필요한 파라미터를 실측/문헌에서 구할 수 있는가 | 1원리 모델은 파라미터 수백 개 필요 |
| 4 | **데이터 요구량** | 보정과 검증에 필요한 실험 데이터의 양 | 순수 경험 모델은 대량 데이터 필요 |
| 5 | **검증 가능성** | 중간 결과를 독립적으로 검증할 수 있는가 | 블랙박스 모델은 검증이 어려움 |
| 6 | **확장성** | 새 물질/조건/모듈을 추가할 수 있는가 | 하드코딩된 경험식은 확장 불가 |
| 7 | **해석 가능성** | 결과의 물리적 의미를 설명할 수 있는가 | 사용자의 도메인 지식 활용 가능성 |

#### 모델 계층의 스펙트럼

```
←── 물리적 충실도 높음          물리적 충실도 낮음 ──→
←── 파라미터 많음              파라미터 적음 ──→
←── 계산 비용 높음              계산 비용 낮음 ──→

1원리 모델        준경험 모델        순수 경험 모델
(Ab Initio)    (Semi-Empirical)    (Empirical)

  예: DFT,         예: 시그모이드       예: 다항식
  양자역학         + 보정항            회귀, 신경망
```

- **1원리 모델**: 기본 물리 법칙에서 출발. 외삽 능력이 강하지만 계산 비용이 높고 파라미터가 많다.
- **준경험 모델**: 물리적 형태를 유지하면서 실험 데이터로 파라미터를 보정. 실용성과 정확도의 균형점.
- **순수 경험 모델**: 데이터로부터 직접 관계를 학습. 보간에는 강하지만 외삽 시 신뢰할 수 없다.

#### 차원 분석과 이론 매뉴얼

모든 방정식에 대해 **차원 분석(Dimensional Analysis)**을 수행하여 단위 일관성을 확인한다. 무차원화는 수치적 안정성을 높이고, 파라미터의 물리적 의미를 명확히 한다.

**이론 매뉴얼(Theory Manual)**은 다음을 포함해야 한다:

- 지배 방정식의 유도 과정
- 모든 가정과 그 근거
- 파라미터 카탈로그: 이름, 기호, 단위, 물리적 의미, 값의 범위, 출처
- 모델의 한계와 적용 범위

#### 민감도 분석 계획

어떤 파라미터가 결과에 가장 큰 영향을 미치는지 사전에 파악하면, 보정(Phase 6) 단계에서 어떤 파라미터를 우선 조정해야 하는지 알 수 있다. 민감도가 높은 파라미터는 정밀한 실험값이 필요하고, 민감도가 낮은 파라미터는 문헌값으로 충분할 수 있다.

### SX 프로젝트 사례

SX 시뮬레이터는 준경험 모델을 선택했다. 완전한 MSE(Mixed Solvent Electrolyte) 열역학 모델 대신, **시그모이드 함수**로 pH-추출률 관계를 근사하는 방식이다:

```
E(pH) = E_max / (1 + exp(-k × (pH - pH50)))
```

이 선택의 근거는 모델 기준 7가지에서 명확하다:
- 파라미터 가용성: 금속당 7개 파라미터(pH50, k, E_max, n_H, n_ext, alpha, C_ref)로 현장 피팅 가능
- 계산 비용: 실시간 UI에서 5단 역류 계산이 1초 이내
- 해석 가능성: pH50은 "추출이 50% 일어나는 pH"라는 직관적 의미

모델 복잡도는 점진적으로 증가했다:
- v1.0: 기본 시그모이드 (Phase 1)
- v1.4: 추출제 로딩 한계 반영 (Phase 2)
- v1.7: 추출제 경쟁 + 수계 종분화 (Phase 3)
- v2.0: 고로딩 다핵착물 방어 (Phase 3 확장)

각 계층 추가는 검증 데이터가 기존 모델의 한계를 드러냈을 때 이루어졌다.

### Quality Gate

- [ ] 모든 지배 방정식이 이론 매뉴얼에 문서화
- [ ] 모든 파라미터에 출처(문헌/실험) 명시
- [ ] 수기 검산(hand calculation) ≥ 1케이스 통과
- [ ] 차원 분석으로 단위 일관성 확인 완료

## 5. Phase 2: 프로토타입과 수기 검산

### 일반론

이 단계의 목적은 "코딩 가능성 확인"이 아니라 **"수학 모델의 올바름 확인"**이다. 코드를 작성하기 전에, 모델이 수학적으로 건전한지 검증한다.

#### MMS(Method of Manufactured Solutions)

MMS는 코드 검증의 황금 표준 중 하나다. 방정식의 해를 인위적으로 구성한 후, 코드가 그 해를 정확하게 재현하는지 확인한다:

1. 원하는 해 형태를 가정한다 (예: u(x) = sin(πx))
2. 이를 지배 방정식에 대입하여 소스 항을 역산한다
3. 코드에 소스 항을 입력하고, 출력이 가정한 해와 일치하는지 확인한다

#### 스프레드시트/스크립트 프로토타입

핵심 방정식 1~2개를 스프레드시트(Excel, Google Sheets)나 간단한 스크립트(MATLAB, Python)로 먼저 구현한다:

- 단일 단계(single stage)만 계산
- 금속 1~2종만 포함
- 경쟁 효과, 로딩 효과 등 복잡한 보정 없이

이 프로토타입으로 수기 검산 결과와 비교하여 엔지니어링 직관을 확인한다. "pH 5에서 Co 추출률이 90%라면 물리적으로 타당한가?" 같은 질문에 답할 수 있어야 한다.

#### 수치적 문제 조기 발견

프로토타입 단계에서 다음을 미리 확인한다:

- **Stiffness**: 시간/공간 스케일이 극단적으로 다른 변수가 공존하는가
- **Ill-conditioning**: 작은 입력 변화가 큰 출력 변화를 유발하는가
- **Singularity**: 특정 입력에서 0으로 나누기 또는 무한대가 발생하는가

#### 해석해가 있는 축소 문제

복잡한 다단 시스템 전체를 검증하기 전에, 해석해(analytical solution)가 알려진 축소 문제로 코드를 검증한다:

- 단일 금속 + 단일 단계 → 분배 계수로 직접 계산 가능
- 무한 단수 극한 → 이론적 분리 효율로 수렴해야 함
- 로딩 없는 희석 조건 → 기본 시그모이드 결과와 일치해야 함

### Quality Gate

- [ ] 프로토타입과 수기 검산의 오차 < 0.1%
- [ ] 해석해가 있는 최소 1개 축소 문제에서 코드 정확성 확인
- [ ] 수치적 위험 요소(stiffness, singularity) 식별 및 대응 방안 문서화

## 6. Phase 3: 핵심 엔진 구현

### 일반론

#### 시뮬레이션 소프트웨어 아키텍처 7원칙

| # | 원칙 | 설명 |
|---|------|------|
| 1 | **계산과 표현의 분리** | 수치 엔진은 UI/시각화 코드를 모른다. `import streamlit`이 엔진에 있으면 안 된다. |
| 2 | **계층적 단방향 의존성** | 파라미터 → 단위 모델 → 유닛 솔버 → 시스템 솔버. 상위 계층이 하위를 호출하되, 역방향 의존은 금지. |
| 3 | **순수 함수 우선** | 동일 입력 → 동일 출력. 전역 상태나 부수 효과를 최소화하여 테스트와 디버깅을 용이하게 한다. |
| 4 | **불변 설정** | 파라미터 딕셔너리를 계산 도중 변경하지 않는다. 깊은 복사(deep copy)로 보호한다. |
| 5 | **Strategy/Plugin 패턴** | 물리 모델을 교체할 수 있도록 인터페이스를 분리한다. 시그모이드 → 다른 모델 전환이 가능해야 한다. |
| 6 | **방어적 수치 계산** | 0으로 나누기, 오버플로우, 음수값에 대해 명시적으로 방어한다. `exp(exponent)` 호출 전 `exponent > 500` 체크. |
| 7 | **검증 가능한 중간 결과** | 각 모듈의 물질수지/에너지수지를 독립적으로 확인할 수 있는 인터페이스를 제공한다. |

#### 모듈 구조 설계

시뮬레이션 엔진의 전형적인 4층 구조:

```
Layer 1: 설정/파라미터 (config)
    ↓
Layer 2: 단위 모델 (isotherm, kinetics, thermodynamics)
    ↓
Layer 3: 유닛 솔버 (single stage, single reactor)
    ↓
Layer 4: 시스템 솔버 (multistage, flowsheet)
```

각 계층은 아래 계층만 호출하고, 자신의 입출력을 명확하게 정의한다. 단위 변환은 계층 경계에서만 수행한다(내부는 SI 단위 또는 일관된 단위 체계 사용).

#### 수렴 반복 구현 시 주의사항

반복 솔버(iterative solver)를 구현할 때 반드시 지켜야 할 사항:

1. **수렴 판정 기준(tolerance)을 명시적으로 정의**: `CONVERGENCE_TOLERANCE = 1e-6`
2. **최대 반복 횟수 제한**: `MAX_ITERATIONS = 500` — 무한 루프 방지
3. **수렴 실패 시 명시적 결과 반환**: `converged=False`를 결과에 포함
4. **Relaxation/damping factor 도입 고려**: 직접 대입(direct substitution)이 발산하면 `x_new = α·x_calc + (1-α)·x_old`

SX 시뮬레이터의 런타임 구조는 교과서적인 4층을 따른다:

## 7. Phase 4: 검증과 확인 (V&V) — 가장 중요한 단계

### 일반론

V&V는 시뮬레이션 소프트웨어 개발에서 **가장 중요한 단계**다. NASA, ASME, DoD가 별도의 표준 문서를 만들어 규정할 만큼, 시뮬레이션의 신뢰성은 V&V 증거의 질과 양에 의해 결정된다.

#### Verification(검증): "방정식을 올바르게 풀고 있는가?"

Verification은 수학/코드의 문제를 다룬다:

**Code Verification**:
- 코드 속 수식이 이론 매뉴얼과 일치하는지 줄 단위 대조
- 단위 변환, 부호, 인덱싱 오류 확인
- "코드가 의도한 수학을 정확하게 구현하는가?"

**Solution Verification**:
- 격자/시간 수렴 연구(Grid/Time Convergence Study): 이산화 오차가 줄어드는지 확인
- Richardson 외삽: 수렴 차수(order of convergence) 확인
- MMS 적용: 인위적으로 구성한 해를 코드가 재현하는지 확인

#### Validation(확인): "올바른 방정식을 풀고 있는가?"

Validation은 물리/모델의 문제를 다룬다:

- 실험 데이터와 시뮬레이션 결과 비교
- **검증 계층** (ASME V&V 10 권장):
  1. 단위 문제(Unit Problem): 개별 물리 모델 검증
  2. 벤치마크(Benchmark): 잘 알려진 표준 문제
  3. 서브시스템(Subsystem): 단일 유닛 수준
  4. 전체 시스템(Complete System): 다단/전체 공정

#### UQ(Uncertainty Quantification, 불확실성 정량화)

불확실성은 두 가지로 구분된다:

- **Aleatory(고유적) 불확실성**: 자연적 변동성. 실험을 아무리 많이 해도 줄어들지 않는다. (예: 원료 조성 변동)
- **Epistemic(인식론적) 불확실성**: 지식 부족에 의한 불확실성. 데이터를 더 모으면 줄어든다. (예: 파라미터 추정 오차)

**Forward UQ**: 입력 불확실성이 출력에 미치는 영향을 정량화한다.
- Monte Carlo 시뮬레이션: 입력 분포에서 샘플링하여 출력 분포를 얻는다
- Latin Hypercube Sampling: 적은 샘플로 효율적인 탐색

**Inverse UQ**: 실험 데이터로부터 파라미터의 불확실성을 역추정한다.
- Bayesian Inference: 사전 분포 + 실험 데이터 → 사후 분포
- Maximum Likelihood Estimation: 실험 데이터를 가장 잘 설명하는 파라미터 추정

#### 자동화된 회귀 테스트

수동 "보고서형" 검증은 한계가 있다. **자동화된 PASS/FAIL 판정 체계**를 구축해야 한다:

```python
# 좋은 예: 자동 판정
assert dataset_mae < THRESHOLD, f"MAE {dataset_mae:.3f} exceeds {THRESHOLD}"

# 나쁜 예: 보고서형
print(f"MAE: {dataset_mae:.3f}")  # 사람이 매번 눈으로 확인해야 함
```

#### 오차 메트릭

| 메트릭 | 수식 | 용도 |
|--------|------|------|
| MAE | Σ\|sim-exp\| / n | 전체적인 절대 오차 수준 파악 |
| RMSE | √(Σ(sim-exp)² / n) | 큰 오차에 높은 가중치 |
| 상대오차 | \|sim-exp\| / \|exp\| × 100% | 스케일이 다른 금속 간 비교 |
| Worst-case | max(\|sim-exp\|) | 최악의 경우 파악 |

**주의**: Trace 수준(매우 낮은 농도)의 금속에 대해 상대오차를 계산하면 분모가 작아 무의미하게 커진다. Trace 기준치(예: 0.1 g/L) 이하는 상대오차 대신 절대오차를 사용해야 한다.

### SX 프로젝트 사례

#### V&V 3단계 구축

SX 시뮬레이터는 두 개의 검증 스크립트를 운영한다:

1. **`validation_test.py`**: 구조 검증 — 단일단 물질수지, Co isotherm 일관성, 다단 역류 물질수지
2. **`test_verification.py`**: 현장 데이터 검증 — Data1~6 프리셋과 시뮬레이션 결과 비교

#### 11.8% 물질수지 오차를 놓친 이유

v1.7.2 이전, 고정 NaOH 모드에서 다단 역류의 수계 누적 유량이 다음 Stage에 전달되지 않아 Ni 물질수지가 11.8% 깨지고 있었다. 이 오류를 한동안 발견하지 못한 이유는 검증이 "보고서형"이었기 때문이다 — 결과를 출력만 하고, 자동으로 FAIL을 판정하지 않았다.

이후 `LEGACY_REGRESSION_MAE_THRESHOLDS`와 `LEGACY_REGRESSION_ABS_ERROR_THRESHOLDS`를 도입하여 자동 판정 체계로 전환했다. 이제 Li MAE > 0.816 g/L이면 자동으로 FAIL이 뜬다.

#### 보고서형→자동판정 전환

```python
# test_verification.py의 자동 판정 구조
def evaluate_legacy_regression(metrics: dict) -> list[str]:
    failures = list(metrics["failures"])
    for dataset in metrics["dataset_metrics"]:
        dataset_mae = dataset.get("mae")
        dataset_threshold = LEGACY_REGRESSION_DATASET_MAE_THRESHOLDS.get(dataset_tag)
        if dataset_mae > dataset_threshold:
            append_unique(failures, f"{dataset_tag}: MAE exceeds threshold")
    return failures
```

- [ ] 금속별 MAE < 사전 정의된 threshold
- [ ] 물질수지 오차 < 0.01%
- [ ] 자동화된 회귀 테스트가 CI/CD 또는 릴리스 전 실행되는 구조 확보

## 8. Phase 5: 사용자 인터페이스와 시각화

### 일반론

#### UI/엔진 철저 분리

시뮬레이션 UI는 반드시 엔진과 분리해야 한다. **MVC(Model-View-Controller)** 또는 **서비스 계층(Service Layer)** 패턴을 사용하여, UI 코드가 수치 엔진을 직접 호출하지 않고 서비스 계층을 통해 간접적으로 사용하도록 한다.

이 분리의 실질적 이점:
- UI를 변경해도 시뮬레이션 결과가 달라지지 않음을 보장
- 같은 엔진을 CLI, 웹앱, API 서버 등 여러 인터페이스에서 재사용 가능
- 엔진의 단위 테스트가 UI 의존성 없이 가능

#### 검증 커버리지 안내 패턴

**모든 시뮬레이션 UI에 필수적인 패턴**: 현재 입력 조건이 검증된 범위 내인지 실시간으로 표시한다.

```
┌──────────────────────────────────────────┐
│ ⚠️ 검증 커버리지: 중간                     │
│ ✅ pH: 5.5 (검증 범위 3.9~7.0 내)          │
│ ✅ 추출제 농도: 0.62M (범위 내)             │
│ ⚠️ 온도: 45°C (검증 범위 20~30°C 밖)       │
│ ❌ 총 금속 농도: 95 g/L (경고 수준 초과)     │
└──────────────────────────────────────────┘
```

사용자가 검증 범위를 벗어난 조건으로 시뮬레이션할 때, 결과의 신뢰성이 낮아질 수 있음을 명확하게 경고해야 한다. "시뮬레이터가 숫자를 출력했으니 맞겠지"라는 과신을 방지하는 핵심 안전장치다.

#### 시각화 종류

시뮬레이션 결과의 해석을 돕는 주요 시각화:

| 시각화 유형 | 용도 | 예시 |
|------------|------|------|
| 프로파일(Profile) | 공간/시간에 따른 변수 변화 | Stage별 pH, 농도 프로파일 |
| 등고선(Contour) | 2D 파라미터 공간에서의 응답 | pH-O/A ratio에 따른 추출률 |
| 수렴 이력 | 솔버의 반복 수렴 과정 | 반복 횟수별 잔차 변화 |
| McCabe-Thiele | 평형선-조작선 시각화 | 다단 추출의 단수 설계 |

#### 수렴 상태 표시

사용자에게 수렴 상태를 항상 표시한다:
- **수렴 여부**: 수렴/미수렴 명시
- **반복 횟수**: 정상 수렴이면 수십 회, 비정상이면 수백 회 → 경고
- **잔차(residual)**: 최종 수렴 오차

#### 입력 검증

물리적으로 불가능한 입력을 차단한다:
- 음수 농도, 음수 유량
- pH 범위 밖의 값
- 0 이하의 추출제 농도
- 물리적으로 비현실적인 조합 (예: 총 금속 > 용해도 한계)

`sx_dashboard.py`는 6차례 리팩토링을 거쳤다. 초기에는 UI 코드와 시뮬레이션 호출이 한 파일에 섞여 있었지만, `dashboard_service.py`(서비스 계층)와 `dashboard_tabs.py`(탭 렌더링)로 분리하면서 역할이 명확해졌다. 현재 구조:

## 9. Phase 6: 현장 보정과 파라미터 튜닝

### 일반론

#### 보정(Calibration) vs 검증(Validation) 구분

**이것은 시뮬레이션 V&V에서 가장 흔한 실수 중 하나다**: 보정에 사용한 데이터로 모델을 검증하면, 과적합(overfitting)을 탐지할 수 없다.

- **보정 데이터**: 파라미터를 조정하는 데 사용
- **검증 데이터**: 보정된 모델의 예측 능력을 평가하는 데 사용
- **두 세트는 반드시 독립적이어야 한다**

데이터가 6세트 있다면, 4세트로 보정하고 2세트로 검증하는 것이 올바른 접근이다.

#### 보정 파라미터 선별

모든 파라미터를 동시에 보정하면 과적합 위험이 높다. **민감도 분석(Phase 1에서 계획)**에 기반하여:

1. 결과에 큰 영향을 미치는 파라미터만 보정 대상으로 선정
2. 나머지는 문헌값 또는 물리적 제약으로 고정
3. 보정 파라미터 수 < 검증 데이터 포인트 수 (자유도 확보)

#### 과적합 방지 체크리스트

- [ ] 보정 데이터와 검증 데이터가 분리되어 있는가?
- [ ] 보정 파라미터 수가 독립 데이터 포인트 수보다 적은가?
- [ ] 보정된 파라미터가 물리적으로 합리적인 범위 내에 있는가?
- [ ] 보정 후 검증 데이터에서의 오차가 보정 데이터와 비슷한 수준인가?

#### 프로필 관리

보정 프로필과 기본값 프로필을 코드 수준에서 분리하여 관리해야 한다:

```python
# 좋은 예: 프로필 분리
LITERATURE_BASE_PARAMS = { ... }      # 문헌 기본값 — 절대 변경하지 않음
SITE_PARAMETER_OVERRIDES = { ... }    # 현장 보정 오버레이
FIELD_CALIBRATED_PARAMS = apply(BASE, OVERRIDES)

# 나쁜 예: 원본 직접 수정
PARAMS["Ni"]["pH50"] = 6.55  # 원래 값(5.8)을 되돌릴 수 없음
```

#### 보정 이력 기록

모든 파라미터 변경에 대해 다음을 기록해야 한다:
- 변경 전 값과 변경 후 값
- 변경 사유 (어떤 데이터가 트리거했는지)
- 영향 범위 (다른 금속/조건에 미치는 영향)
- 변경 일시와 변경자

### SX 프로젝트 사례

SX 시뮬레이터의 `config.py`는 `LITERATURE_BASE_PARAMS`와 `SITE_PARAMETER_OVERRIDES`를 명확히 분리한다:

```python
SITE_PARAMETER_OVERRIDES = {
    ("Cyanex 272", "Ni"): {"pH50": 6.55, "k": 2.0},   # 문헌 5.8 → 현장 6.55
    ("Cyanex 272", "Li"): {"pH50": 8.10, "k": 1.8},
    ("D2EHPA", "Li"):     {"pH50": 6.3},
    ("D2EHPA", "Co"):     {"E_max": 99.0},
    ("D2EHPA", "Mg"):     {"pH50": 3.2},
}
```

이 구조 덕분에 사용자는 사이드바에서 `현장 보정 (Isd 108)`과 `문헌 기본값` 프로필을 전환하여 비교할 수 있다.

그러나 한 가지 중요한 교훈이 있다: **한 금속의 파라미터 변경이 경쟁 추출을 통해 다른 금속에 영향을 미친다.** Ni의 pH50을 5.8에서 6.55로 올리면, Ni의 추출이 늦어지면서 같은 pH에서 Co와 경쟁하는 양상이 바뀐다. 경쟁 추출 모델에서는 파라미터를 한 금속씩 독립적으로 보정할 수 없으며, 전체 금속을 동시에 고려해야 한다.

- [ ] 문헌 기본값으로 복원하는 기능이 동작
- [ ] 보정된 파라미터가 물리적으로 합리적인 범위 내

## 10. Phase 7: 견고성과 극한 조건 처리

### 일반론

#### 스트레스 테스트

입력 범위의 **경계와 그 너머**에서 시뮬레이터가 어떻게 동작하는지 확인한다:

- 극단적으로 높은/낮은 농도
- 극단적인 pH (< 1 또는 > 12)
- O/A ratio 극값 (0.01 또는 100)
- 모든 금속 농도가 0인 경우
- 추출제가 완전히 소진된 경우

#### 도메인 지식 기반 방어

극한 조건에서의 문제를 해결할 때, **수치적 트릭보다 물리적 메커니즘 이해**가 더 효과적이다.

수치적 트릭의 예:
```python
# 위험: 물리적 의미 없는 클램핑
result = max(0, min(100, raw_result))
```

도메인 지식 기반 방어의 예:
```python
# 좋은 예: 물리적 메커니즘을 반영
# 고로딩 시 다핵착물 형성으로 유효 화학양론이 감소하는 현상을 모델링
n_eff = n_ext * max(0.35, 1.0 - loading_ratio**2)
```

#### 수치 안정성 방어 체크리스트

| 위험 | 방어 | 예시 |
|------|------|------|
| 0으로 나누기 | `max(denominator, 1e-30)` 또는 분기 처리 | D 계산 시 E ≈ 100% |
| 지수 오버플로우 | exponent 클램핑 | `if exponent > 500: return 0.0` |
| 음수 농도 | 물질수지에서 `max(0, C)` 적용 | 반복 수렴 중 발진 |
| 극단 D값 | 상한/하한 설정 | `D = max(1e-6, min(1e6, D))` |
| 로딩률 > 100% | 물리적 상한 적용 | `min(1.0, loading_ratio)` |

#### Graceful Degradation

수렴 실패 시 시뮬레이션을 완전히 중단하기보다, **경고와 함께 최선 추정치를 제공**하는 것이 사용자에게 더 유용하다:

```python
result = {
    "converged": False,
    "iterations": MAX_ITERATIONS,
    "warning": "최대 반복 횟수 도달. 결과는 근사값입니다.",
    "raffinate": best_estimate,  # 마지막 반복의 결과
}
```

#### 새로운 물리 메커니즘의 발견

극한 조건 테스트는 종종 기존 모델이 포착하지 못하는 **새로운 물리 메커니즘**을 드러낸다. 이때 수치적 패치가 아닌 모델 확장이 필요한지 판단해야 한다:

- 패치로 충분: 기존 메커니즘의 수치적 안정성 문제 (오버플로우, 발산)
- 모델 확장 필요: 기존 메커니즘이 포착하지 못하는 새로운 물리 현상

### SX 프로젝트 사례 3건

#### 사례 1: 물질수지 1줄 수정 (11.8% → 0.002%)

v1.7.2에서 발견된 물질수지 버그는 **코드 1줄**의 문제였다. 고정 NaOH 모드에서 Stage별로 NaOH가 투입될 때 수계 누적 유량(`Q_aq_cumulative`)이 다음 Stage에 전달되지 않았다. 극한 조건(높은 NaOH 투입량)에서만 오차가 두드러져, 일반 조건 테스트에서는 놓쳤다.

**교훈**: 물질수지는 모든 경로에서 자동 검증되어야 한다.

#### 사례 2: 다핵착물 알고리즘 (39 g/L → 0.1 g/L 오차)

Data5(Ni 71 g/L)는 추출제가 극도로 부족한 고농도 조건이다. 기존 경쟁 추출 모델은 Ni를 전혀 추출하지 못하는 "과억제" 오류를 보였다(시뮬레이션 Ni 후액 ~55 g/L vs 실험 16 g/L).

이 문제는 수치적 트릭으로 해결할 수 없었다. **문헌 조사** 결과, D2EHPA와 Cyanex 272는 추출제 부족 환경에서 다핵 올리고머(multi-nuclear oligomeric complexes)를 형성하여 금속당 필요한 추출제가 감소하는 현상이 알려져 있었다. 이를 n_eff 동적 감쇠 알고리즘으로 구현:

```python
n_eff_multiplier = max(0.35, 1.0 - (loading_ratio_clamped ** 2))
```

결과: Ni 후액 오차 39 g/L → 0.1 g/L로 극적 개선.

**교훈**: 극한 조건의 실패는 새로운 물리를 배울 기회다.

#### 사례 3: 적응형 Relaxation (500회 미수렴 → 62회 수렴)

다단 역류 solver가 고정 NaOH 모드에서 500회 반복 후에도 수렴하지 않았다. 원인은 직접 대입(direct substitution)의 발산이었다. 4단계 적응형 relaxation factor(0.2→0.5→0.7→0.9)를 도입하여 62회 만에 수렴을 달성했다.

**교훈**: 수렴 알고리즘은 처음부터 적응형으로 설계해야 한다.

## 11. Phase 8: 리팩토링과 유지보수

### 일반론

#### 기술 부채 인벤토리

정기적으로(매 3~5개 릴리스마다) 기술 부채를 목록화하고 우선순위를 매긴다:

| 부채 항목 | 위험도 | 해결 비용 | 우선순위 |
|-----------|--------|-----------|---------|
| 전역 상태 의존 | 높음 | 중간 | 1 |

| 보고서형 검증 | 높음 | 낮음 | 1 |
| 하드코딩된 매직 넘버 | 중간 | 낮음 | 3 |

#### 리팩토링 안전 프로토콜

시뮬레이션 코드 리팩토링은 일반 코드보다 훨씬 위험하다. 수식의 미세한 변경이 결과를 완전히 바꿀 수 있기 때문이다. 안전 프로토콜:

1. **전체 검증 실행** — 리팩토링 전 모든 테스트 PASS 확인
2. **한 관심사만 분리** — 한 번에 하나의 리팩토링만 수행
3. **재검증** — 리팩토링 후 동일 테스트 PASS 확인 + 수치 결과 비트 수준 동일성 확인
4. **기록** — 무엇을 왜 바꿨는지, 무엇을 의도적으로 바꾸지 않았는지 문서화

특히 **"의도적으로 하지 않은 일"**을 기록하는 것이 중요하다. 이는 다음 리팩토링 담당자에게 "이것은 실수로 빠진 게 아니라 의도적으로 미룬 것"이라는 신호를 보낸다.

#### ADR(Architecture Decision Records)

주요 아키텍처 결정을 짧은 문서로 기록한다:

```
# ADR-003: 전역 EXTRACTANT_PARAMS 대신 명시적 파라미터 전달

## 상태: 제안됨 (2026-03-08)

## 맥락
EXTRACTANT_PARAMS 전역 딕셔너리가 여러 모듈에서 참조되어,
파라미터 프로필 전환 시 예측 불가능한 부작용 발생

## 결정
ParameterSet 또는 SimulationConfig 객체를 solver에 명시적으로 전달

## 결과
- (+) 테스트에서 파라미터 격리 가능
- (+) 동시에 여러 프로필로 시뮬레이션 실행 가능
- (-) 모든 함수 시그니처에 파라미터 인자 추가 필요
```

#### "새 기능 추가" vs "품질 유지"의 긴장 관리

시뮬레이션 프로젝트에서 가장 흔한 함정은 기능을 계속 추가하면서 검증과 리팩토링을 미루는 것이다. **Feature freeze 기간**을 주기적으로 도입하여:

- 새 기능 추가 금지
- 기존 코드의 테스트 커버리지 확대
- 기술 부채 해소
- 문서화 보완

#### 계층적 문서화

```
Level 1: 코드 내 docstring
  → 함수 단위의 입출력, 수식, 가정

Level 2: 모듈 설명 (파일 상단 docstring)
  → 모듈의 역할, 다른 모듈과의 관계

Level 3: 프로젝트 구조 문서
  → PROJECT_STRUCTURE.md — 런타임 흐름, 계층 구조, 읽는 순서 추천

Level 4: 변경 이력
  → CHANGELOG.md (사용자 기능), 리팩토링 기록 (내부 구조)
```

#### "신규 개발자 30분 테스트"

새로운 사람이 프로젝트에 합류했을 때, **30분 내에 다음을 파악**할 수 있어야 한다:

1. 이 프로젝트가 무엇을 하는 소프트웨어인가?
2. 어떤 파일을 어떤 순서로 읽어야 하는가?
3. 어떻게 실행하는가?
4. 어떤 테스트를 어떻게 돌리는가?
5. 현재 알려진 한계는 무엇인가?

30분 내에 이것이 불가능하면 문서화가 부족한 것이다.

## 부록 A: 개발 안티패턴

### 1. "동작하니까 배포"

**문제**: 시뮬레이션이 숫자를 출력하면 맞다고 가정한다.
**위험**: 결과가 "그럴듯하게 틀릴 수 있다". 추출률 85%라는 숫자가 나왔다고 해서 그것이 물리적으로 맞다는 보장은 없다.
**대안**: V&V를 통과하지 않은 결과는 "미검증"으로 명시한다.

### 2. "전부 한 파일에"

**문제**: 모든 코드가 한 파일에 집중되어 파일이 1,000줄을 넘긴다.
**SX 사례**: `single_stage.py`가 1,036줄. 목표 pH/고정 NaOH/사포니피케이션 모드가 혼재.
**대안**: 관심사 분리. 모드별 Strategy 패턴 또는 최소한 helper 함수 추출.

### 3. "수식은 머릿속에"

**문제**: 이론 매뉴얼 없이 코드에만 수식이 존재한다.
**위험**: 원개발자가 떠나면 수식의 가정과 한계를 아무도 모른다.
**대안**: Phase 1에서 이론 매뉴얼을 코드보다 먼저 작성한다.

### 4. "테스트는 나중에"

**문제**: 보고서형 검증만 하고 자동 PASS/FAIL 판정을 미룬다.
**SX 사례**: 11.8% 물질수지 오차를 한동안 발견하지 못함.
**대안**: 첫 번째 검증 데이터가 확보되는 즉시 자동 판정 체계를 구축한다.

### 5. "파라미터 하드코딩"

**문제**: `0.5`, `2.0`, `1e-6` 같은 매직 넘버가 코드 곳곳에 산재한다.
**위험**: 같은 물리 상수가 여러 곳에 다른 값으로 존재할 수 있다.
**대안**: 모든 파라미터를 중앙 설정 모듈에서 관리한다.

### 6. "보정 데이터로 검증"

**문제**: 파라미터를 맞춘 데이터로 모델을 검증한다고 주장한다.
**위험**: 과적합. 보정 데이터에는 완벽하지만 새로운 조건에서 실패한다.
**대안**: 보정/검증 데이터를 코드 수준에서 분리하고, 교차 검증을 수행한다.

### 7. "한 번에 모든 기능"

**문제**: 경쟁 추출, 종분화, 로딩, 온도 효과를 한 번에 전부 구현한다.
**위험**: 버그가 어디서 발생하는지 격리할 수 없다.
**SX 사례**: v1.0(기본) → v1.4(로딩) → v1.7(경쟁+종분화) → v2.0(다핵착물) 순서로 점진적 추가.
**대안**: 하나를 추가하고, 검증하고, 안정화한 후 다음을 추가한다.

### 8. "전역 상태 의존"

**문제**: 파라미터가 전역 변수로 암묵적으로 전달된다.
**SX 사례**: `EXTRACTANT_PARAMS` 전역 딕셔너리를 `extraction_isotherm.py`가 직접 import.
**위험**: 테스트에서 파라미터를 격리할 수 없고, 동시 실행 시 충돌 가능.
**대안**: 파라미터를 함수 인자로 명시적으로 전달한다.

### 9. "수렴하면 끝"

**문제**: solver가 수렴했으니 결과가 올바르다고 가정한다.
**위험**: 국소 최소(local minimum)에 수렴했을 수 있다. 물리적으로 틀린 해에 수렴할 수 있다.
**대안**: 수렴 후에도 물질수지/에너지수지를 독립적으로 확인한다.

### 10. "범위 밖 외삽"

**문제**: 검증된 조건 범위를 벗어난 예측을 신뢰한다.
**SX 사례**: Data5(Ni 71 g/L)는 기존 검증 범위를 크게 초과. 모델이 완전히 새로운 물리를 요구.
**대안**: 검증 범위를 UI에 표시하고, 범위 밖에서는 명시적 경고를 발행한다.

## 부록 B: Quality Gate 종합표

| Phase | Quality Gate | 측정 방법 |
|-------|-------------|-----------|
| **0. 문제 정의** | 검증 데이터 ≥ 3세트 + 정확도 기준 숫자화 + 범위 정의 | 문서 검토 |
| **1. 모델 설계** | 모든 방정식 문서화 + 파라미터 출처 + 수기 검산 ≥ 1케이스 | 이론 매뉴얼 대조 |
| **2. 프로토타입** | 프로토타입-수기 검산 오차 < 0.1% | 수치 비교 |
| **3. 엔진 구현** | 단일 유닛 물질수지 < 1e-4 + 수기 검산 일치 | 자동 테스트 |
| **4. V&V** | 모든 데이터셋 수렴 + 메트릭 < threshold + 물질수지 < 0.01% | 자동 회귀 테스트 |
| **5. UI/시각화** | UI 변경이 시뮬레이션 결과에 영향 없음 | 회귀 테스트 |
| **6. 보정** | 보정/검증 데이터 분리 + 보정 이력 기록 | 문서 + 코드 검토 |
| **7. 견고성** | 극한 입력에서 크래시 없음 + 적절한 경고 | 스트레스 테스트 |
| **8. 리팩토링** | 전후 수치 동일 + 모든 테스트 PASS | 비트 수준 비교 |
| **9. 문서화** | 4대 문서 존재 + 30분 테스트 통과 | 동료 검토 |

## 부록 C: 시뮬레이션 소프트웨어 아키텍처 7원칙

1. **계산과 표현의 분리 (Model-View Separation)**
   - 수치 엔진은 UI 프레임워크를 import하지 않는다
   - 같은 엔진을 CLI, 웹앱, API에서 재사용 가능

2. **계층적 단방향 의존성 (Layered One-Way Dependency)**
   - 파라미터 → 단위 모델 → 유닛 솔버 → 시스템 솔버
   - 하위 계층은 상위 계층의 존재를 모른다

3. **순수 함수 우선 (Pure Functions First)**
   - 동일 입력 → 동일 출력
   - 전역 상태나 부수 효과를 최소화

4. **불변 설정 (Immutable Configuration)**
   - 파라미터는 계산 시작 시 깊은 복사(deep copy)로 보호
   - 런타임 중 파라미터 변경 금지

5. **Strategy/Plugin 패턴 (Model Swappability)**
   - 물리 모델을 교체 가능하도록 인터페이스 분리
   - 시그모이드 → Langmuir → NRTL 등 전환 가능

6. **방어적 수치 계산 (Defensive Numerics)**
   - 모든 나눗셈에 0 체크
   - 지수 함수에 오버플로우 방어
   - 물리적 범위 제약 적용 (음수 농도 방지)

7. **검증 가능한 중간 결과 (Verifiable Intermediates)**
   - 각 모듈이 물질수지/에너지수지를 독립적으로 보고
   - 중간 결과를 외부에서 확인할 수 있는 인터페이스 제공
