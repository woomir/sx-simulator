# v2.1.1 리팩토링 기록

작성일: 2026-03-08

## 1. 목적

이번 리팩토링의 목적은 웹앱의 사용자 기능이나 시뮬레이션 결과를 바꾸지 않고,
코드 중복과 데이터 드리프트 위험을 줄여 이후 수정과 확장을 더 안전하게 만드는 것이다.

이번 작업은 **백엔드/내부 구조 정리** 성격이므로 `CHANGELOG.md`에는 반영하지 않았다.

## 2. 원칙

- 사용자 UI 동작과 시뮬레이션 결과를 바꾸지 않는다.
- 현장 프리셋, 검증 스크립트, 대시보드가 서로 다른 데이터를 따로 들고 있지 않게 만든다.
- 수식 자체는 유지하고, 중복된 계산 루프만 helper로 정리한다.
- 리팩토링 후에는 반드시 기존 검증 스크립트와 앱 기동으로 회귀 확인을 한다.

## 3. 이번 라운드에서 실제로 한 일

### 3.1 공용 데이터 모듈 신설

신규 파일:

- `sx_simulator/datasets.py`

이 모듈로 아래 항목을 옮겼다.

- Data1~Data6 현장 데이터 정의
- 프리셋 주의사항 (`KNOWN_PRESET_NOTES`)
- 검증 제외 포인트 (`KNOWN_INVALID_POINTS`)
- 총 황산염 농도 계산식 (`calc_sulfate_from_feed`)

의도:

- 대시보드 프리셋과 검증 스크립트가 같은 원본 데이터를 사용하게 함
- 프리셋 조건과 검증 기대값이 따로 놀지 않게 함
- 황산염 계산 로직이 대시보드와 검증 스크립트에서 다시 갈라지지 않게 함

### 3.2 대시보드 프리셋/황산염 계산 연결 정리

수정 파일:

- `sx_dashboard.py`

변경 내용:

- 로컬에 하드코딩돼 있던 `PRESETS` 정의 제거
- `sx_simulator.datasets`에서 `DASHBOARD_PRESETS`를 가져오도록 변경
- 로컬 `KNOWN_PRESET_NOTES` 정의 제거
- 황산염 농도 계산을 공용 helper `calc_sulfate_from_feed()`로 교체

효과:

- 프리셋 데이터와 검증 데이터가 동일 출처를 사용하게 됨
- 이후 Data1~Data6 수정 시 한 군데만 바꾸면 됨

### 3.3 검증 스크립트의 데이터 원본 통합

수정 파일:

- `test_verification.py`

변경 내용:

- 실험 데이터 리스트와 예외 포인트 정의를 제거
- `sx_simulator.datasets`의 `VERIFICATION_EXPERIMENTS`, `KNOWN_INVALID_POINTS`,
  `calc_sulfate_from_feed()`를 사용하도록 변경

효과:

- 대시보드와 검증 스크립트가 동일한 현장 데이터 기준을 사용함
- 이름 형식까지 기존 출력과 동일하게 유지하도록 구성함

### 3.4 single-stage 내부 계산 루프 정리

수정 파일:

- `sx_simulator/single_stage.py`

추가된 내부 helper:

- `_partition_stage_with_damping()`
- `_solve_competitive_stage_state()`

정리한 중복:

- 목표 pH 모드와 고정 NaOH 모드에서 반복되던 금속별 분배 계산
- 경쟁 추출 모드 결과를 stage state로 바꾸는 후처리
- H+ 방출량 계산 루프

중요한 점:

- 화학식이나 수지식 자체는 바꾸지 않았다.
- target pH 모드와 fixed NaOH 모드의 기존 반복 패턴 차이도 그대로 유지했다.
  - target pH 모드: 기존 damping 업데이트/종료 방식 유지
  - fixed NaOH 모드: 기존 내부 loading 반복 후 최종 재계산 방식 유지

## 4. 의도적으로 하지 않은 일

이번 라운드에서는 아래는 건드리지 않았다.

- `EXTRACTANT_PARAMS` 전역 상태 주입 구조 제거
- `sx_dashboard.py`의 탭별 모듈 분리
- `multistage_sx.py`의 stage pass 중복 제거
- 검증 스크립트의 PASS/FAIL 자동 판정 체계 추가

이유:

- 위 항목들은 효과는 크지만 범위가 커서, 이번처럼 "동작 동일성 보장"이 중요한 라운드에서는 위험도가 높다고 판단했다.
- 이번 라운드는 먼저 **저위험 리팩토링**만 적용했다.

## 6. 다음 리팩토링 우선순위

다음 라운드에서 추천하는 순서는 아래와 같다.

1. `EXTRACTANT_PARAMS` 전역 변경 구조 제거
   - `ParameterSet` 또는 `SimulationConfig` 객체를 solver에 명시적으로 전달
2. `sx_dashboard.py`를 입력/실행/탭 렌더링 단위로 분리
3. `multistage_sx.py`의 stage pass 로직 helper 추출
4. 검증 스크립트를 자동 판정형 회귀 테스트로 전환

## 7. 참고

이번 리팩토링은 사용자 기능 추가가 아니라 내부 구조 정리다.
따라서 이후 사용자 커뮤니케이션이 필요할 때는 `CHANGELOG.md`보다 이 문서를 우선 참고하는 편이 맞다.
