# v2.1.1 시뮬레이션 개선 누적 로그

이 문서는 시뮬레이션 엔진 관련 변경을 단계별로 누적 기록한다.

원칙:

- 한 번에 한 가지 축만 변경한다.
- 각 단계마다 의도와 범위를 명확히 남긴다.
- 계산 결과가 의도적으로 바뀌는 단계인지, 구조 정리만 하는 단계인지 구분한다.
- 각 단계의 검증 결과는 별도 문서 `v2.1.1_시뮬레이션_검증_누적로그.md`에 누적한다.

---

## 1단계. 파라미터 체계 분리

### 목적

현 시뮬레이터는 `field_calibrated`와 `literature_default` 프로필을 지원하지만, 코드 구조상 현장 보정값이 기본 딕셔너리에 섞여 있어 일반화 범위를 판단하기 어려웠다.

이번 단계의 목적은 아래 두 가지다.

- 문헌 기본값과 현장 보정값을 코드상에서 명확히 분리
- 런타임 기본 동작은 유지하여 시뮬레이션 결과를 바꾸지 않음

### 변경 파일

- `sx_simulator/config.py`

### 변경 내용

1. `LITERATURE_BASE_PARAMS`를 신설했다.
   - 문헌/기본 추정값을 보관하는 원본 파라미터 세트다.
2. `SITE_PARAMETER_OVERRIDES`를 신설했다.
   - 현장 보정값은 override로 별도 관리한다.
3. `_build_parameter_profile()` helper를 추가했다.
   - 기본 파라미터와 override를 조합해 프로필을 생성한다.
4. `FIELD_CALIBRATED_PARAMS`는 이제 `LITERATURE_BASE_PARAMS + SITE_PARAMETER_OVERRIDES`로 생성된다.
5. `EXTRACTANT_PARAMS`는 하위 호환용 기본 별칭으로 유지했다.
   - 엔진이 명시적 파라미터 세트를 받지 않을 때의 기본 동작은 기존과 동일하다.
6. `get_parameter_profile("literature_default")`는 이제 현장 보정값을 되돌리는 방식이 아니라, 문헌 기본값 복사본을 직접 반환한다.

### 현재 프로필 차이

이번 단계에서 코드상으로 명시된 현장 오버레이는 아래 두 항목이다.

- `Cyanex 272 / Ni / pH50`: `5.8 -> 6.3`
- `D2EHPA / Mg / pH50`: `4.5 -> 3.2`

### 기대 효과

- 앞으로 어떤 값이 범용 기본값이고 어떤 값이 현장 보정값인지 즉시 파악 가능
- 다음 단계에서 `site-specific` 보정 범위를 체계적으로 확장 가능
- 일반화 성능 검토 시 프로필 차이의 출처를 코드에서 직접 추적 가능

### 비의도적 변경 방지 기준

이번 단계는 모델 구조 개선이 아니라 파라미터 계층 구조 정리다.

- 시뮬레이션 기본 결과는 유지되어야 한다.
- 현장 검증 오차 패턴은 이전과 동일해야 한다.
- `validation_test.py`의 구조 검증은 그대로 통과해야 한다.

### 다음 단계 후보

2단계는 아래 둘 중 하나가 유력하다.

- `K_MSO4`를 실제 계산에 연결하는 황산염 착물/자유 금속 보정
- `목표 pH` 모드의 NaOH 희석 self-consistent 보정

정확도 개선 효과는 황산염 착물 반영 쪽이 더 클 가능성이 높다.

## 2단계. 황산염 착물/자유 금속 speciation 연결

### 목적

`config.py`에는 `K_MSO4`가 정의되어 있었지만, 실제 계산에서는 거의 사용되지 않았다.

이번 단계의 목적은 아래와 같다.

- 황산(HSO4-) 평형과 별개로 금속 황산염 착물(`MSO4^0`) 효과를 코드에 연결
- 자유 금속 분율이 낮아지면 hydrolysis(`MOH+`)도 줄어들도록 반영
- 아직 추출 D 자체는 건드리지 않고, 먼저 speciation/pH submodel을 정합화

### 변경 파일

- `sx_simulator/extraction_isotherm.py`
- `sx_simulator/single_stage.py`

### 변경 내용

1. `get_free_sulfate_concentration()` 추가
   - 총 황산염 농도와 pH로부터 자유 `SO4^2-` 농도를 계산한다.
2. `get_aqueous_speciation_state()` 추가
   - 단순 1:1 가정에서 금속별 `free / hydroxo / sulfate` 종분율을 계산한다.
3. `calc_aq_protons()` 개선
   - 기존에는 `K_MOH`만 사용해 hydrolysis를 계산했다.
   - 이제 `K_MSO4`까지 반영한 종분율에서 `hydroxo_fraction`을 직접 사용한다.
   - 즉, 황산염 착물이 강한 조건에서는 자유 금속이 줄고, 그에 따라 `MOH+` 생성도 줄어든다.

### 이번 단계에서 일부러 하지 않은 것

이번 단계에서는 아래를 의도적으로 보류했다.

- `distribution_coefficient()` 직접 보정
- `compute_competitive_extractions()`의 D 계산에 sulfate factor 삽입

이유는 현재 추출 모델이 경험식 `pH50/k/E_max` 기반이라, 황산염 보정을 바로 D에 곱하면 기존 현장 보정값과 중복될 가능성이 크기 때문이다.

즉 이번 단계는 먼저 `speciation submodel`을 물리적으로 맞추는 작업이다.

### 기대 효과

- 고황산염 조건에서 자유 금속 분율과 hydrolysis 계산의 물리적 일관성 향상
- 이후 단계에서 `free metal` 기반 추출 보정을 넣을 수 있는 기반 확보
- `K_MSO4`가 설정만 존재하는 죽은 상수 상태 해소

### 한계

이번 단계만으로는 현장 후액 농도 오차가 바로 개선되지 않을 수 있다.

특히 `test_verification.py`는 대부분 `목표 pH` 모드에서 돌아가므로, 이번 단계의 영향은 주로 `NaOH/pH submodel`과 고정 NaOH 모드 쪽에서 먼저 나타난다.

### 다음 단계 후보

3단계는 아래 두 방향 중 하나다.

- `free metal fraction`을 추출 D에 상대 보정 형태로 연결
- `목표 pH` 모드에서 NaOH 희석 self-consistent 반복 도입

정량 정확도 개선만 보면 첫 번째가 더 직접적이고, 구조적 일관성만 보면 두 번째가 더 중요하다.

## 3단계. D2EHPA-Co 고황산염 상대 보정

### 목적

2단계에서 `K_MSO4`를 speciation 계산에 연결했지만, 현장 후액 농도는 거의 변하지 않았다.

그 이유는 현재 추출 모델이 경험식 `pH50/k/E_max` 기반이라, sulfate 효과를 모든 금속에 일괄 곱하면 기존 현장 보정값과 충돌하기 때문이다.

이번 단계의 목적은 아래와 같다.

- 전 금속 일괄 보정이 아니라, 실제 검증에서 문제로 드러난 조합만 선택 보정
- D2EHPA 고황산염 조건에서 Co 과추출을 줄여 현장 데이터 정합도 개선
- Co 로딩 완화에 따른 Ni 거동 개선이 있는지 함께 확인

### 변경 파일

- `sx_simulator/config.py`
- `sx_simulator/extraction_isotherm.py`
- `sx_simulator/single_stage.py`

### 변경 내용

1. `SULFATE_D_CORRECTION_RULES` 추가
   - 현재는 `("D2EHPA", "Co")` 조합만 활성화
2. `get_sulfate_d_correction_factor()` 추가
   - 2단계에서 만든 상대 자유 금속 계수를 기반으로 D 보정 계수를 생성
   - 규칙이 없는 금속/추출제 조합은 보정하지 않음
3. D 적용 위치
   - 일반 분배 경로 `_partition_stage_with_damping()`
   - 경쟁 추출 경로 `compute_competitive_extractions()`
4. 현재 적용식
   - `raw_factor = relative_free_metal_factor`
   - `correction = clamp(raw_factor ** 4, 0.2, 1.0)`

### 설계 이유

- 초기 시도처럼 모든 금속에 sulfate 상대 보정을 적용하면 Ni 오차가 크게 악화됐다.
- 반면 D2EHPA-Co만 선택적으로 억제하면:
  - Co 과추출이 줄고
  - 유기상 로딩 경쟁이 완화되어
  - 일부 고황산염 조건에서 Ni도 같이 개선됐다.

즉 이번 단계는 `전체 물리모델 일반화`보다 `현장 오차 패턴을 망치지 않는 범위의 선택 보정`에 가깝다.

### 현재 보정 거동

예시:

- `D2EHPA / Co / sulfate 0.7 M` → 보정 계수 `1.0`
- `D2EHPA / Co / sulfate 1.0 M` → 보정 계수 `1.0`
- `D2EHPA / Co / sulfate 1.5 M` → 보정 계수 약 `0.316`
- `Cyanex 272 / Co / sulfate 1.5 M` → 보정 없음 (`1.0`)
- `D2EHPA / Ni / sulfate 1.5 M` → 보정 없음 (`1.0`)

### 기대 효과

- Data4~6 같은 고황산염 D2EHPA 조건에서 Co 후액 예측 개선
- 과도한 Co 우선 추출 억제로 Ni 후액 예측도 일부 안정화
- 2단계의 speciation 계산을 실제 추출 계산으로 연결하는 첫 번째 제한적 적용 사례 확보

### 한계

- 아직 일반화된 sulfate 추출 보정 모델은 아니다.
- D2EHPA-Co 외 조합에는 동일한 방식이 유효한지 검증되지 않았다.
- 따라서 이 단계는 `선택적 보정`이며, 이후에는 metal/extractant별 독립 검증이 더 필요하다.

### 다음 단계 후보

4단계는 아래 두 방향 중 하나다.

- `target pH` 모드의 희석/알칼리 계산 구조를 정리
- 검증 스크립트를 허용오차 기반 자동 회귀 테스트로 강화

지금 시점에서는 엔진 정합도 개선과 함께 검증 자동화를 병행하는 편이 안전하다.

## 4단계. Cyanex 272-Ni 현장 재보정

### 목적

3단계까지 진행한 뒤에도 가장 큰 잔여 오차는 Cyanex 272 조건의 Ni 과추출이었다.

- `Data1 Ni`: `-6.397 g/L`
- `Data2 Ni`: `-2.996 g/L`

반면 `Data3`는 이미 충분히 맞고 있었다.

즉 문제는 `Ni 전체 모델이 틀렸다`기보다, 현장 프로필의 `Cyanex 272 / Ni` 전이 위치와 전이 폭이 저 pH 구간에서 아직 너무 공격적이라는 쪽에 가까웠다.

이번 단계의 목적은 아래와 같다.

- 별도 임시 보정식을 추가하지 않고
- 현장 오버레이 파라미터 자체를 다시 맞춰
- `Data1/2`를 개선하면서 `Data3`를 유지

### 변경 파일

- `sx_simulator/config.py`

### 변경 내용

`SITE_PARAMETER_OVERRIDES`의 `Cyanex 272 / Ni` 항목을 아래처럼 조정했다.

- `pH50`: `6.3 -> 6.55`
- `k`: `2.5 -> 2.0`

### 설계 이유

사전 grid 확인 결과, 저 pH Ni 과추출 문제는 별도 선택 보정보다 `pH50 + k` 재보정으로 더 깔끔하게 줄어들었다.

핵심 관찰:

- `pH50`만 올리면 `Data1`은 좋아지지만 `Data2`가 빠르게 과보정됐다.
- `k`를 함께 낮추면 `6.1`과 `5.89` 사이 전이 폭이 완만해져 `Data1/2`를 동시에 더 잘 맞출 수 있었다.
- `Data3`는 `pH 7.0`이라 여전히 충분히 높은 추출 영역에 있어 영향이 거의 없었다.

즉 이번 단계는 `새 알고리즘 추가`가 아니라, `현장 프로필의 Ni 전이 위치/기울기 재보정`이다.

### 기대 효과

- Cyanex 272 저 pH 조건에서 Ni 과추출 완화
- 고 pH Cyanex 272 조건(`Data3`) 유지
- 선택 보정 규칙 추가 없이 현장 프로필의 일관성 향상

### 한계

- 이번 단계는 여전히 `field_calibrated` 프로필 개선이다.
- 문헌 기본값(`literature_default`)의 일반화를 직접 개선하는 단계는 아니다.
- 따라서 다른 희석제나 다른 Cyanex 272 계 현장에 바로 일반화된다고 보기는 어렵다.

### 다음 단계 후보

5단계는 아래 두 방향 중 하나가 적절하다.

- `target pH` 모드의 NaOH 희석 self-consistent 계산 도입
- `test_verification.py`를 허용오차 기반 회귀 테스트로 강화

## 5단계. 목표 pH 모드 NaOH 희석 self-consistent 계산

### 목적

기존 `target pH` 모드는 목표 pH에서 금속 평형을 계산한 뒤 필요한 NaOH 몰수만 역산했고, 그에 따른 수계 유량 증가와 희석 효과는 반영하지 못했다.

이 구조는 특히 다음 상황에서 한계가 있었다.

- 높은 목표 pH
- 알칼리 요구량이 큰 고황산염/고금속 조건
- target pH 결과를 실제 고정 NaOH 투입 조건과 대응시켜 보고 싶은 경우

이번 단계의 목적은 아래와 같다.

- `target pH` 모드에서 선택적으로 NaOH 희석 효과까지 self-consistent하게 반영
- 기존 현장 프리셋/검증 흐름은 깨지지 않도록 기본 동작은 유지
- 새 경로가 실제 `fixed NaOH` 계산과 서로 일관되는지 검증 가능하게 만들기

### 변경 파일

- `sx_simulator/single_stage.py`
- `sx_simulator/multistage_sx.py`
- `sx_simulator/dashboard_service.py`
- `sx_dashboard.py`
- `validation_test.py`
- `docs/user_manual.md`

### 변경 내용

1. `single_stage.py`
   - 목표 pH 평형 계산을 별도 helper로 분리했다.
   - `C_NaOH > 0`인 경우에만 NaOH 유량을 반복 역산해 `Q_aq_out` 증가를 self-consistent하게 반영하도록 추가했다.
   - 결과에 `Q_NaOH_estimated_L_hr`, `Q_aq_out_L_hr`, `dilution_iterations`를 함께 담도록 했다.
2. `multistage_sx.py`
   - 목표 pH 모드에서 stage별 자동 NaOH 유량이 생기면 다음 stage의 수계 유량에 누적 반영하도록 수정했다.
   - 결과에 `NaOH_flow_profile`을 추가했다.
3. `dashboard_service.py`
   - 목표 pH 모드에서도 `C_NaOH`를 선택적으로 엔진에 전달하도록 수정했다.
   - 검증 커버리지 메시지에 `희석 추정 활성` 안내를 추가했다.
4. `sx_dashboard.py`
   - 목표 pH 모드에 `NaOH 희석 self-consistent 추정` 옵션을 추가했다.
   - 사용자가 이 옵션을 켠 경우에만 `NaOH 농도 (M, 희석 추정용)`을 입력받는다.
   - 기본값은 OFF로 두어 기존 현장 프리셋/검증과의 일관성을 유지했다.
5. `validation_test.py`
   - `target pH + C_NaOH`로 얻은 자동 NaOH 유량을 `fixed NaOH`에 재주입했을 때 pH와 후액이 거의 동일한지 확인하는 synthetic consistency test를 추가했다.

### 설계 이유

- 현장 프리셋(Data1~6)은 기존 엔진 한계를 우회하기 위해 이미 `희석된 입력 농도 + 합산 수계 유량` 형태로 저장돼 있다.
- 따라서 self-consistent 희석을 기본 ON으로 바꾸면 기존 검증 데이터는 이중 희석이 된다.
- 이번 단계에서는 **기본 동작 유지**를 우선하고, 새 계산 경로는 `C_NaOH`를 명시한 경우에만 활성화하도록 설계했다.

즉 이번 단계는 `legacy compatibility`를 유지한 채, 더 물리적인 target pH 경로를 별도로 추가한 것이다.

### 기대 효과

- 사용자가 target pH 모드에서 NaOH 농도까지 알고 있을 때, 희석을 포함한 더 현실적인 후액 예측 가능
- target pH와 fixed NaOH 모드를 직접 대응시켜 비교 가능
- 이후 현장 데이터 구조를 raw feed 기준으로 재정의할 때 엔진 쪽 기반이 이미 준비됨

### 한계

- 기본 OFF이므로, 현재 현장 프리셋 검증 수치는 이번 단계에서 의도적으로 바꾸지 않았다.
- NaOH 농도 가정이 잘못되면 희석 예측도 달라진다.
- Data1~6에는 실제 NaOH 농도 정보가 없어서, 이번 단계만으로 field MAE가 자동 개선되지는 않는다.

### 다음 단계 후보

6단계는 아래 두 방향이 유력하다.

- 현장 검증 데이터를 `raw feed + 실제/가정 NaOH 조건` 기준으로 재구성
- `test_verification.py`를 허용오차 기반 회귀 테스트로 강화

## 6단계. 현장 검증 데이터 basis 재구성

### 목적

5단계까지의 엔진은 `target pH + NaOH 희석 self-consistent` 경로를 지원하게 됐지만, 현장 검증 데이터(Data1~6)는 여전히 `희석된 입력 농도 + 합산 수계 유량` 기준으로 검증되고 있었다.

즉 현재 검증 체계에는 두 가지 basis가 혼재해 있었다.

- **legacy premixed basis**:
  - NaOH가 이미 섞인 농도와 유량을 엔진 입력으로 사용
- **raw feed basis**:
  - 원수 농도와 feed flow를 그대로 사용하고
  - NaOH 유량/농도는 엔진이 직접 처리

이번 단계의 목적은 아래와 같다.

- 검증 스크립트가 두 basis를 명확히 구분해서 다룰 수 있게 정리
- raw-feed 검증이 가능하도록 엔진의 황산염 농도도 유량 변화에 따라 함께 희석 반영
- 어떤 오차가 `모델 자체` 때문이고 어떤 오차가 `NaOH 조건 가정` 때문인지 분리해서 볼 수 있게 만들기

### 변경 파일

- `sx_simulator/single_stage.py`
- `sx_simulator/multistage_sx.py`
- `sx_simulator/datasets.py`
- `test_verification.py`

### 변경 내용

1. `single_stage.py`
   - `dilute_concentration_by_flow()` helper를 추가했다.
   - target pH / fixed NaOH 모드 모두에서 `C_sulfate`를 유량 증가에 맞춰 희석한 뒤 speciation과 proton balance에 사용하도록 수정했다.
   - 결과에 `C_sulfate_out_M`를 담아 다음 stage로 전달할 수 있게 했다.
2. `multistage_sx.py`
   - stage 진행 시 `C_sulfate_current`를 함께 업데이트하도록 변경했다.
   - 즉 NaOH가 누적 투입되면 sulfate 농도도 stage별로 함께 희석된다.
3. `datasets.py`
   - `VALIDATION_BASES`, `DEFAULT_ASSUMED_NAOH_CONCENTRATION_M`를 추가했다.
   - `dilute_feed_by_flow()`와 `prepare_verification_case()`를 추가해
     - `legacy_premixed_target_pH`
     - `raw_feed_target_pH`
     두 basis를 공통 helper로 준비하도록 정리했다.
4. `test_verification.py`
   - 이제 두 basis를 모두 실행한다.
   - raw-feed basis에서는 가정 NaOH 농도와 데이터의 `naoh_flow` 대비 엔진이 추정한 총 NaOH 유량도 함께 출력한다.

### 설계 이유

- raw-feed basis를 돌리려면 단순히 입력 농도만 바꾸는 것으로는 부족했다.
- 기존 엔진은 NaOH에 의한 유량 증가를 반영하면서도 sulfate 농도는 고정된 상수처럼 취급하고 있었기 때문에, raw-feed 기준에서는 내부 농도장이 물리적으로 어긋날 수 있었다.
- 따라서 이번 단계는 단순 `검증 스크립트 수정`이 아니라, **유량 증가에 따른 sulfate 희석 전파**까지 포함한 검증 인프라 정리다.

### 기대 효과

- legacy premixed 기준의 기존 정합도와 raw-feed 기준의 가정 민감도를 같은 스크립트에서 비교 가능
- 앞으로 실제 NaOH 농도 정보가 들어오면 raw-feed 기준 검증으로 자연스럽게 전환 가능
- `회귀 테스트`를 만들 때 어느 basis를 고정할지 명시적으로 선택 가능

### 한계

- 현재 Data1~6에는 실제 NaOH 농도 정보가 없다.
- 따라서 raw-feed basis는 이번 단계에서도 `5.0 M` 가정값 기반의 preview 성격이다.
- 즉 raw-feed 결과가 곧바로 더 “정답에 가깝다”는 뜻은 아니다.

### 다음 단계 후보

7단계는 아래 두 방향 중 하나다.

- `test_verification.py`를 허용오차 기반 회귀 테스트로 강화
- Data1~6 또는 신규 현장 데이터에 실제 NaOH 농도 메타데이터를 부여해 raw-feed basis를 정식 주 검증 경로로 승격

## 7단계. 허용오차 기반 회귀 테스트 고정

### 목적

6단계까지 오면서 검증 스크립트는 두 가지 basis를 모두 다룰 수 있게 됐지만, 여전히 출력 리포트 중심이어서 모델이 나빠져도 배포 전에 자동으로 막히지 않는 상태였다.

이번 단계의 목적은 다음과 같다.

- 현재 가장 안정적인 `legacy_premixed_target_pH`를 공식 회귀 기준으로 고정
- 금속별 MAE, dataset별 MAE, 핵심 포인트별 절대오차에 허용오차를 부여
- `raw_feed_target_pH`는 아직 NaOH 메타데이터가 부족하므로 진단용으로만 유지

즉 이번 단계는 **검증을 문서 출력에서 pass/fail 체계로 승격**하는 작업이다.

### 변경 파일

- `sx_simulator/datasets.py`
- `test_verification.py`

### 변경 내용

1. `datasets.py`
   - `LEGACY_REGRESSION_MAE_THRESHOLDS`를 추가했다.
   - `LEGACY_REGRESSION_DATASET_MAE_THRESHOLDS`를 추가했다.
   - `LEGACY_REGRESSION_ABS_ERROR_THRESHOLDS`를 추가했다.
   - 즉 허용오차 기준을 코드 안에 분산시키지 않고 dataset 정의와 같은 곳에서 관리하도록 정리했다.
2. `test_verification.py`
   - basis별 실행 결과를 구조화된 metrics로 수집하도록 수정했다.
   - dataset별로 `converged`, `iterations`, `dataset_MAE`, `worst error`, `metal errors`를 저장한다.
   - `evaluate_legacy_regression()`를 추가해 legacy basis만 pass/fail 평가에 사용하도록 했다.
   - 조건을 어기면 `SystemExit(1)`로 종료해 CI/배포 전 자동 차단이 가능해졌다.
   - raw-feed basis는 계속 출력하지만, 최종 상태 판정에는 반영하지 않는다.

### 설계 이유

- 현재 field validation에서 가장 일관적인 기준은 아직 `legacy premixed basis`다.
- raw-feed basis는 엔진상 가능하지만, Data1~6의 실제 NaOH 농도 정보가 없어 가정값 민감도가 너무 크다.
- 따라서 지금 단계에서 raw-feed까지 강제 pass/fail에 넣으면, 모델 회귀보다 메타데이터 부족이 더 큰 실패 원인이 된다.

즉 이번 단계는 **현재 신뢰할 수 있는 검증 축만 먼저 고정**한 것이다.

### 기대 효과

- 모델 성능이 의도치 않게 악화되면 `test_verification.py`가 즉시 실패한다.
- 다음 단계에서 파라미터나 화학 모델을 조정할 때, 최소한 현재 legacy field 성능은 자동으로 보호된다.
- raw-feed basis는 별도 진단값으로 계속 보존되므로, 장기적으로 주 검증 기준을 옮길 여지도 남겨둔다.

### 한계

- 허용오차는 현재 성능 수준을 기준으로 설정한 것이다. 즉 “문헌상 절대 진실”이 아니라 “회귀 방지용 guardrail”에 가깝다.
- raw-feed basis는 아직 진단용이므로, 실제 NaOH 농도 메타데이터가 확보되기 전까지는 정식 pass/fail 축이 아니다.

### 다음 단계 후보

- Data1~6 또는 신규 현장 데이터에 실제 NaOH 농도 메타데이터를 추가해 raw-feed basis를 승격
- 활동도/이온강도 또는 자유금속 기반 경쟁추출 보정을 추가해 legacy 허용오차 자체를 더 낮추기

## 8단계. 다단 솔버 수치 안정화

### 목적

7단계까지의 검증 체계는 field regression을 안정적으로 보호할 수 있게 됐지만, 종합 stress 검증에서는 `random multistage` 조건에서 다단 수렴 실패가 다수 확인됐다.

이 문제는 연구원 추가 데이터 없이도 먼저 손볼 수 있는 영역이었다.

이번 단계의 목적은 다음과 같다.

- field regression 정확도는 유지
- 단일단/현장 보정 범위는 그대로 보존
- 다단 fixed-point 반복의 수치 안정성을 개선
- 특히 `target pH + 다단 + 고황산염` 조합에서의 불필요한 solver 붕괴를 줄이기

즉 이번 단계는 **화학모델 수정이 아니라 수치해석/솔버 구조 개선**이다.

### 변경 파일

- `sx_simulator/multistage_sx.py`
- `docs/v2.1.1_다음단계_연구원협의_기획.md`
- `docs/v2.1.1_시뮬레이션_종합검증_보고서.md`

### 변경 내용

1. `multistage_sx.py`
   - NaOH 분배 로직을 helper로 분리했다.
   - outer iteration에 adaptive relaxation을 도입했다.
   - best iterate를 추적해, 미수렴 시에도 마지막 iterate가 아니라 가장 안정적이었던 iterate를 기반으로 최종 exact pass를 수행하도록 바꿨다.
   - final exact pass를 1회가 아니라 여러 번 self-consistent refinement 하도록 보강했다.
   - target pH 다단 해가 직접 fixed-point로 안 잡히는 경우, `target continuation` fallback을 추가했다.
   - fallback에서도 더 나은 residual을 보인 해를 선택하도록 정리했다.
2. 연구원 협의 문서
   - 이번 단계에서 실제로 내가 바로 진행 가능한 부분은 `수치 안정화`이고,
   - 여전히 연구원 판단이 필요한 부분은 `raw-feed 메타데이터`와 `화학모델 일반화`라는 점을 분리해서 유지했다.
3. 종합 검증 보고서
   - 안정화 적용 전/후 stress 결과를 반영할 수 있도록 후속 결과를 기록했다.

### 설계 이유

- 기존 다단 엔진은 현장 검증 범위에서는 잘 맞았지만, 외삽 조건에서 fixed-point coupling이 강해지면 마지막 iterate가 그대로 붕괴된 상태로 최종 결과에 반영될 수 있었다.
- 특히 target pH 모드는 single-stage 안쪽 dilution loop와 multistage 바깥쪽 counter-current loop가 동시에 얽혀 있어, direct iteration만으로는 민감도가 커질 수 있다.
- 따라서 이번 단계는 기존 해법을 버리지 않고:
  - direct solve 유지
  - 실패 시 continuation fallback
  - final exact refinement 강화
  순으로 “보수적 보강”을 적용했다.

### 기대 효과

- field validation 결과를 바꾸지 않으면서 multistage robustness 일부 개선
- random multistage failure의 구조를 더 좁혀서, 남는 문제를 `수치 문제`와 `모델 문제`로 구분하기 쉬워짐
- 이후 연구원 협의가 필요한 화학모델 개선과 독립적으로 solver 품질을 개선 가능

### 한계

- D2EHPA 고황산염 target pH 다단 조건은 여전히 취약하다.
- 즉 이번 단계만으로 “임의 다단 운전영역 전체 robust” 수준이 되지는 않았다.
- 남은 문제는 일부 수치해석 이슈이기도 하지만, 일부는 model envelope 바깥 조건일 가능성도 있다.

### 다음 단계 후보

- 연구원과 협의해 `운전영역 밖`인지 `실제 개선 대상`인지 먼저 판단
- 그 판단 이후
  - raw-feed 메타데이터 정비
  - D2EHPA 고황산염 target pH 모델 일반화
  - 혹은 multistage residual 재정의
  순서로 이어가는 것이 맞다.
